# 3.23 — Achievements and Reputation

**Status:** Draft for review.  
**Owner:** Karluiz + Covenant.  
**Product home:** Metrics / Covenant Score.  
**Implementation shape:** umbrella spec — break into smaller execution specs before handing to AOM.

## 1. Goal

Add a long-lasting achievements system to Covenant that celebrates trustworthy autonomous work: reliable operators, disciplined orchestration, safe execution, project learning, and verified outcomes.

Achievements should feel like GitHub-style badges, but they must reward quality signals rather than raw usage volume. The system lives primarily inside **Metrics**, with small contextual badge summaries in Operator and Orchestrator surfaces.

## 2. Product thesis

Covenant is not only a terminal. It is becoming a local AI workbench where operators, orchestrators, executors, project memory, and the user collaborate over time.

A good achievement system should answer:

- **Can I trust this operator?**
- **Is the orchestrator getting better at coordinating work?**
- **Does this project have stronger memory and rituals than before?**
- **Did Covenant protect me from risky behavior?**
- **Are we shipping verified work, not just producing activity?**

The north star:

> Achievements celebrate durable behaviors that make Covenant safer, more reliable, and more useful over months of real work.

## 3. Principles

1. **Quality over volume.** Never reward "run 1,000 commands" or "send 500 prompts" by itself. Reward verified completion, safety, recovery, project understanding, and good engineering rituals.
2. **Local-first and private.** Achievements derive from local metric facts. No raw terminal output, raw ANSI, secrets, or unmasked prompt contents are stored for achievement evaluation.
3. **Cosmetic first.** Achievements, badges, titles, and reputation do not unlock dangerous autonomy or bypass safety policies.
4. **Long arcs, not dopamine loops.** Prefer tiered progress over months to disposable daily streaks.
5. **Explainable.** Each earned badge must show why it was earned using safe metadata: repo, operator, task id, verification kind, timestamp, and non-sensitive counts.
6. **Operator and orchestrator identity.** Operators can earn reliability/craft badges; the orchestrator can earn coordination/recovery badges; projects can earn memory/mastery badges.
7. **No bad incentives.** Hidden/rare badges must not encourage risky commands, unnecessary token spend, or fake failures.
8. **Reviewable and resettable.** The user can inspect, hide, export, or reset local achievements.

## 4. Core concepts

### 4.1 Achievement

A durable rule that can be earned once or across tiers.

Example: `clean_run` is earned when a task completes with verification and no failed commands in that task window.

### 4.2 Badge

The visual representation of an earned achievement tier.

Example: **Clean Run III** with a silver/gold/platinum ring.

### 4.3 Title

A cosmetic label derived from a high-tier achievement or a reputation dimension.

Examples:

- `Guardian`
- `Cartographer`
- `Reliable Finisher`
- `Conductor`
- `Recovery Artist`

Titles can be shown beside an Operator or Orchestrator label, but they must not change permissions.

### 4.4 Reputation

A rollup profile made from achievements across dimensions:

| Dimension | Meaning |
|---|---|
| **Craft** | Verified, tested, documented, low-noise engineering work. |
| **Safety** | Risky actions handled correctly; secrets protected. |
| **Reliability** | Work finishes, resumes, and recovers from failure. |
| **Orchestration** | Task splitting, delegation, conflict avoidance, budget discipline. |
| **Memory** | Project notes, learned commands, reusable knowledge, summaries. |
| **Focus** | Useful autonomy with minimal unnecessary interruptions. |

Reputation is a display/summary layer only. It should be calculated from earned achievement tiers and visible in Metrics.

### 4.5 Subject

The entity receiving credit.

| Subject | Examples |
|---|---|
| `operator` | A named Operator persona or task slot owner. |
| `orchestrator` | The central coordinator / teammate runtime / AOM coordinator. |
| `project` | A repo root or workspace. |
| `user` | Global user-level achievements across Covenant. |
| `system` | Safety/system achievements that are not owned by one operator. |

### 4.6 Scope

Where progress is counted.

| Scope | Use |
|---|---|
| `global` | Across all Covenant usage. |
| `repo` | Project-specific mastery. |
| `operator` | Per Operator persona. |
| `orchestrator` | Per orchestrator profile. |
| `session` | Only for transient detail, not long-term badge identity. |

Operator rows may show a compact badge/title:


- Show at most 1 title + 1 small badge by default.
- Full details link back to Metrics.
- Do not crowd the live operator controls.

The orchestrator surface may show:

```text
Orchestrator · Conductor II · Recovery Artist I · Budget Keeper I
```

For multi-task views, show a tiny badge only after a mission completes, not during active work.

### 5.4 Celebration behavior

Default should be subtle:

- New badge toast after task completion or session end, never while the terminal is actively waiting for input.
- Toast contains badge name, tier, and "View in Metrics".
- Settings can disable celebrations entirely.
- Hidden achievements reveal only when earned.

## 6. Achievement tiers and rarity

### 6.1 Tier shape

Most achievements use five tiers:

| Tier | Label | Default target style |
|---|---|---|
| I | Bronze | first meaningful event |
| II | Silver | repeated behavior |
| III | Gold | strong habit |
| IV | Platinum | long-term mastery |
| V | Diamond | exceptional long arc |

Default count targets should be selected per badge type:

| Pattern | Targets | Use |
|---|---:|---|
| Common habit | `1, 5, 25, 100, 500` | verified completions, summaries, docs-first runs |
| Hard habit | `1, 3, 10, 25, 100` | recoveries, conflict avoidance, budget wins |
| Rare habit | `1, 2, 5, 10, 25` | crash recovery, hidden badges, multi-session arcs |
| Project mastery | `1, 3, 10, 30, 100` | per-repo learned commands, notes, verified tasks |

### 6.2 Rarity

Rarity describes how unusual the badge is, not how important it is.

| Rarity | Meaning |
|---|---|
| Common | Expected healthy behavior. |
| Uncommon | Requires several good runs. |
| Rare | Requires recovery, coordination, or stronger evidence. |
| Epic | Long-term, multi-session, or high-quality work. |
| Legendary | Hidden or very uncommon, but never dangerous. |

### 6.3 Reputation points

Each earned tier contributes points to one or more dimensions. Points are only used to render summary bars/titles.

Suggested default:

| Tier | Points |
|---|---:|
| I | 10 |
| II | 25 |
| III | 60 |
| IV | 150 |
| V | 400 |

If an achievement spans multiple dimensions, split the points evenly or declare explicit weights in the definition.

## 7. Verification levels

Achievements should reason about **verified work**, not self-reported success.

```text
none          — no verification signal
self_report   — executor/operator said it is done, but no external evidence
command_passed — relevant test/lint/build/check command passed
user_accepted — user accepted/confirmed the outcome
commit_observed — commit scanner observed a commit for the task/repo
release_event  — release/build artifact or release workflow observed
```

Rules:

- Craft achievements require at least `command_passed`, `user_accepted`, or `commit_observed` unless explicitly marked as non-code.
- Operator reliability achievements may use `user_accepted` for review/watch tasks where tests are irrelevant.
- Safety achievements can be earned from policy events without a task completion.
- The UI should display the verification level on detail pages.

## 8. Metrics event model

### 8.1 Current foundation

Covenant Score already records local-first metric data in `crates/score` and surfaces it in `ui/src/score`:

- prompts
- commits
- LLM calls / token usage
- specs
- repo / branch / group attribution
- optional sync

Achievements should extend this layer instead of creating a second telemetry system.

### 8.2 New achievement facts

Add normalized achievement facts for events that are too specific for the existing `score_events` table.

A fact is not an achievement. It is safe metadata from which achievements are computed.

Examples:

- `task_completed`
- `task_verified`
- `operator_decision_persisted`
- `operator_escalated`
- `orchestrator_plan_created`
- `orchestrator_task_delegated`
- `task_recovered`
- `risky_action_confirmed`
- `risky_action_blocked`
- `secret_redacted`
- `docs_read_before_edit`
- `project_note_created`
- `project_note_reused`
- `project_command_learned`
- `test_command_passed`
- `build_command_passed`
- `lint_command_passed`
- `session_resumed`
- `task_scope_conflict_avoided`

### 8.3 Fact schema

Proposed SQLite table in `score.sqlite`:

```sql
CREATE TABLE IF NOT EXISTS achievement_facts (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  ts_ms          INTEGER NOT NULL,
  day            TEXT NOT NULL,
  kind           TEXT NOT NULL,

  subject_type   TEXT NOT NULL, -- operator | orchestrator | project | user | system
  subject_id     TEXT,

  repo           TEXT,
  branch         TEXT,
  group_name     TEXT,
  session_id     TEXT,
  task_id        TEXT,

  verification   TEXT,          -- none | self_report | command_passed | user_accepted | commit_observed | release_event
  dedupe_key     TEXT UNIQUE,
  metadata_json  TEXT NOT NULL DEFAULT '{}'
);

CREATE INDEX IF NOT EXISTS idx_achievement_facts_kind
  ON achievement_facts(kind, ts_ms);
CREATE INDEX IF NOT EXISTS idx_achievement_facts_subject
  ON achievement_facts(subject_type, subject_id, ts_ms);
CREATE INDEX IF NOT EXISTS idx_achievement_facts_repo
  ON achievement_facts(repo, ts_ms);
CREATE INDEX IF NOT EXISTS idx_achievement_facts_task
  ON achievement_facts(task_id, kind);
```

`metadata_json` must contain only allowlisted keys. Examples:

```json
{
  "command_kind": "test",
  "exit_code": 0,
  "files_changed": 4,
  "operator_id": "athena",
  "risk_kind": "destructive_command",
  "verification": "command_passed"
}
```

Forbidden metadata:

- raw terminal output
- raw prompts
- raw command bodies for risky commands unless sanitized and explicitly allowed
- secrets or secret-like values
- raw ANSI
- full file contents

### 8.4 Progress and award schema

Progress can be recomputed from facts, but earned awards should be persisted so the user has a stable timeline.

```sql
CREATE TABLE IF NOT EXISTS achievement_progress (
  achievement_id TEXT NOT NULL,
  subject_type   TEXT NOT NULL,
  subject_id     TEXT,
  subject_key    TEXT NOT NULL DEFAULT '', -- normalized subject_id for stable PKs
  scope_type     TEXT NOT NULL,            -- global | repo | operator | orchestrator
  scope_id       TEXT,
  scope_key      TEXT NOT NULL DEFAULT '', -- normalized scope_id for stable PKs
  tier           INTEGER NOT NULL DEFAULT 0,
  progress       INTEGER NOT NULL DEFAULT 0,
  target         INTEGER NOT NULL DEFAULT 0,
  updated_at_ms  INTEGER NOT NULL,
  PRIMARY KEY (achievement_id, subject_type, subject_key, scope_type, scope_key)
);
```

`subject_id` / `scope_id` are nullable for API ergonomics. `subject_key` /
`scope_key` are the non-null database keys used to avoid SQLite `NULL`
uniqueness surprises.

```sql
CREATE TABLE IF NOT EXISTS achievement_awards (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  achievement_id TEXT NOT NULL,
  tier           INTEGER NOT NULL,
  title          TEXT NOT NULL,
  subject_type   TEXT NOT NULL,
  subject_id     TEXT,
  subject_key    TEXT NOT NULL DEFAULT '',
  scope_type     TEXT NOT NULL,
  scope_id       TEXT,
  scope_key      TEXT NOT NULL DEFAULT '',
  repo           TEXT,
  branch         TEXT,
  earned_at_ms   INTEGER NOT NULL,
  seen_at_ms     INTEGER,
  details_json   TEXT NOT NULL DEFAULT '{}',
  UNIQUE (achievement_id, tier, subject_type, subject_key, scope_type, scope_key)
);
```

### 8.5 Server sync policy

Covenant already has a deployed `covenant-server` and Metrics sync path, so
achievements can become a cross-device identity layer instead of staying
trapped on one machine. This is valuable, but it should ship in privacy tiers.

Recommended rollout:

#### Tier 0 — Local-only MVP

- Achievement facts, progress, and awards stay in local `score.sqlite`.
- This proves the rule engine and UI without adding server risk.

#### Tier 1 — Sync earned awards only

Sync only stable, safe award rows:

- `achievement_id`
- `tier`
- `subject_type`
- `subject_id` or safe display alias
- `scope_type`
- `repo` display key or hash, depending on Metrics privacy settings
- `earned_at_ms`
- `details_json` after allowlisting

Do **not** sync raw facts yet. This lets a user keep badges across machines
without uploading detailed behavioral history.

#### Tier 2 — Sync sanitized progress aggregates

Sync aggregate counters needed for cross-device progress:

- progress count toward next tier;
- current tier;
- last updated timestamp;
- reputation dimension totals.

This avoids sending detailed event streams while still allowing "Clean Run
17/25" to continue on another device.

#### Tier 3 — Optional fact sync for advanced/server features

Only if needed later, sync allowlisted facts. This unlocks server-side
recompute, global rarity stats, and richer cross-device audit trails, but it
has higher privacy and schema-stability cost.

Even at Tier 3, sync must never include:

- raw command output;
- raw terminal text;
- raw prompts;
- raw model responses;
- raw ANSI;
- secrets or secret-like strings;
- full file contents.

#### Server-side use cases

A server-backed achievement profile could enable:

- cross-device badge continuity;
- restore after reinstall;
- global rarity percentages like GitHub achievements;
- optional shareable profile cards;
- release/community achievements;
- team/org-level aggregate achievements later.

#### Sync principles

- Achievement sync is opt-in through the existing Metrics/GitHub account
  flow.
- The local database remains source of truth while offline.
- Server rows are append-friendly and versioned.
- Award uniqueness is enforced by `(user_id, achievement_id, tier,
  subject_key, scope_key)`.
- Conflict resolution is monotonic: highest tier wins, earliest
  `earned_at_ms` is preserved, progress keeps the max known value unless a
  full recompute is requested.
- The UI should clearly label whether achievements are `Local` or `Synced`.

## 9. Achievement engine

### 9.1 Location

Proposed module layout:

| Path | Role |
|---|---|
| `crates/score/src/achievements.rs` | Definitions, rule engine, progress calculation, award generation. |
| `crates/score/src/achievement_types.rs` | Serializable types for Tauri/UI. Optional if `achievements.rs` grows large. |
| `crates/score/src/store.rs` | SQLite migrations and query methods. |
| `crates/app/src/score_commands.rs` | Tauri command wrappers. |
| `ui/src/score/achievements.ts` | Metrics achievement UI. |
| `ui/src/score/achievementTypes.ts` | TS types if the API grows large. |
| `ui/src/score/styles.css` | Badge grid and reputation bars. |

### 9.2 Definition shape

Rust-side static definitions should be the source of truth.

```rust
pub struct AchievementDefinition {
    pub id: &'static str,
    pub title: &'static str,
    pub summary: &'static str,
    pub category: AchievementCategory,
    pub rarity: AchievementRarity,
    pub subject: AchievementSubjectKind,
    pub scope: AchievementScopeKind,
    pub hidden: bool,
    pub tiers: &'static [AchievementTier],
    pub reputation: &'static [ReputationWeight],
    pub rule: AchievementRule,
}

pub struct AchievementTier {
    pub tier: u8,
    pub label: &'static str,
    pub target: u32,
    pub title_unlocked: Option<&'static str>,
}
```

The UI should receive serialized definitions so copy and targets do not drift.

### 9.3 Rule engine

Rules should be pure and deterministic:

1. Record fact with idempotent `dedupe_key`.
2. Evaluate affected achievement definitions.
3. Update progress rows.
4. Insert award rows for newly crossed tiers.
5. Emit a frontend event for unseen awards.

For expensive rules, run a debounced recompute:

- immediate evaluation for simple count facts;
- periodic/on-open recompute for sequence rules like "docs read before edit" or "red to green";
- manual `achievement_recompute` command for development/debugging.

### 9.4 Dedupe strategy

Every fact emitter must provide a stable `dedupe_key` when possible:

```text
<kind>:<task_id>:<subject_id>:<semantic_id>
```

Examples:

- `task_completed:task_123:athena`
- `test_command_passed:session_abc:block_42`
- `secret_redacted:session_abc:event_99`
- `project_note_created:/repo/path:note_id`

If no stable key exists, use a conservative generated key and avoid awarding tier progress from noisy facts.

## 12. Achievement detail page copy

Each achievement detail should include:

- Name, icon, rarity, tier.
- One-sentence purpose.
- Progress to next tier.
- Subject and scope.
- Verification requirements.
- Safe recent evidence.
- Reputation dimensions affected.

Example:

```text
Clean Run II
Completed 5 verified tasks without failed command blocks.

Subject: Athena
Scope: Operator
Progress: 5 / 25 toward Clean Run III
Verification: command_passed or user_accepted
Recent evidence: karlTerminal · task 82 · tests passed · 2026-05-24
```

## 13. Privacy and safety requirements

- Do not persist raw terminal text for achievements.
- Do not persist raw ANSI.
- Do not persist raw prompts or raw model responses.
- Do not persist secrets or suspected secrets.
- Risky command achievements store `risk_kind` and decision outcome, not full command text by default.
- Project paths should follow existing Metrics privacy policy; prefer repo display names over absolute paths in UI.
- Sync is opt-in and must follow Metrics sync controls.
- Achievement facts must be safe to export in a support bundle.

## 14. Anti-gaming rules

Avoid achievements for:

- number of prompts sent;
- number of commands run;
- tokens spent;
- time spent in app;
- causing failures just to recover them;
- bypassing safety prompts;
- high-risk destructive operations.

Use safeguards:

- Idempotent dedupe keys.
- Require verification for craft/reliability badges.
- Require task ids/session ids for task-scoped badges.
- Rate-limit facts that can be emitted repeatedly from the same block/session.
- For recovery badges, require an actual blocked/failed state followed by successful completion, not just a failed command.
- For project notes, count unique note ids and later add user/helpfulness signals.

## 15. Settings

Add a compact settings section under Metrics settings:

| Setting | Default | Meaning |
|---|---:|---|
| `metrics.achievements.enabled` | `true` | Record local achievement facts and awards. |
| `metrics.achievements.celebrations` | `true` | Show subtle earned badge toasts. |
| `metrics.achievements.hidden_badges` | `true` | Allow hidden badges to be revealed when earned. |
| `metrics.achievements.include_project_paths` | existing Metrics policy | Whether repo/project labels use full paths or display names. |
| `metrics.achievements.reset` | action | Delete local facts/progress/awards after confirmation. |

If Metrics itself is disabled in a future privacy mode, achievements should also pause.

## 19. Acceptance criteria for the full system

### Metrics / storage

- [ ] Achievement facts are stored in `score.sqlite` with repo/branch/group/session/task attribution when available.
- [ ] Fact metadata is allowlisted and contains no raw terminal output, raw ANSI, raw prompts, raw model responses, or secrets.
- [ ] Achievements can be recomputed from facts without losing already-earned award timeline entries.
- [ ] Duplicate facts do not duplicate progress or awards.
- [ ] Existing Score summary/heatmap/usage behavior is unchanged.

### Rule engine

- [ ] Static achievement definitions serialize to the frontend.
- [ ] Count-based tiered achievements update when relevant facts are recorded.
- [ ] Newly crossed tiers insert award rows exactly once.
- [ ] Progress shows next tier target or completed state.
- [ ] Hidden achievements are invisible until earned unless debug mode is enabled.

### UI

- [ ] Metrics contains an Achievements section with Summary, In Progress, Earned, Operators, Orchestrator, Projects, and Recent Awards views or filters.
- [ ] Achievement details explain why the badge was earned using safe evidence.
- [ ] Earned badge toasts are subtle, optional, and link to Metrics.
- [ ] Operator panel can show a compact title/badge without crowding controls.
- [ ] Orchestrator/teammate surfaces can show compact badge summaries.

### Product behavior

- [ ] No achievement rewards raw command count, raw prompt count, token spend, or unsafe behavior.
- [ ] Craft/reliability achievements require verification where applicable.
- [ ] Safety achievements are generated only from central safety/secret-handling paths.
- [ ] Achievements remain cosmetic and do not affect permissions, autonomy, or safety bypasses.

### Tests

- [ ] Rust tests cover tier crossing, dedupe, hidden badge visibility, award uniqueness, and recompute.
- [ ] Rust tests cover privacy metadata allowlisting for achievement facts.
- [ ] UI tests cover rendering earned/in-progress states and hidden badge behavior.
- [ ] Regression tests confirm existing Metrics commands and UI still work.

## 22. Open questions for review

1. **Name of the system:** `Achievements`, `Reputation`, `Badges`, `Covenant Mastery`, or something more branded?
2. **Default visibility:** Should achievements be enabled by default because they are local-only, or gated behind Metrics opt-in?
3. **Celebrations:** Do we want GitHub-like badge popovers, or a very quiet toast only after task/session completion?
4. **Reputation wording:** Should we call the rollup `Reputation`, `Mastery`, `Trust`, or `Discipline`?
5. **Operator titles:** Should titles be auto-selected only, or can the user pin a favorite title per operator?
6. **Project scope key:** Should repo achievements be keyed by absolute repo path, git remote URL, or display name + path hash?
7. **Sync tier:** Should the first server-backed version sync only earned awards, or also sanitized progress aggregates?
8. **User feedback:** Should users be able to mark an escalation/result as helpful, enabling higher-quality achievements later?
9. **Rare badges:** Do hidden badges fit Covenant's tone, or should everything be transparent?
10. **MVP catalog:** Which 8–12 badges should ship first?
