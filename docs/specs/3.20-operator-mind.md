# OperatorMind — Per-Tab Persistent Agent State

**Status:** Design approved (2026-05-06). Implementation pending.
**Branch:** `feature/operator-mind`
**Author:** Karluiz + Claude Opus 4.7
**Supersedes:** Stateless polling operator (`crates/app/src/operator.rs` decide-and-act loop).

---

## 1. Problem

The current operator feels dumb because it has no memory of its own work. Every poll is a fresh inference: it sees the tail of PTY bytes, a static persona/mission, and semantic recall hits, then emits one of four actions and forgets. This causes:

- **Apparent amnesia.** It re-asks the same question, re-tries the same failed approach, doesn't notice it just acted 30 seconds ago.
- **Tunnel vision.** A 16KB tail window is the entire context per turn; long-running deliberations scroll off-screen between polls.
- **No self-coherence.** It cannot say "I planned to X next, but Y happened, so I'm doing Z instead." There is no plan to refer to.
- **State-blindness.** A yes/no prompt, a diff review, and a stuck loop are all just "tail" to it; same prompt, same disposition.

Most of what feels like "the operator is dumb" is actually "the operator has no memory." This spec replaces stateless polling with a thread-aware agent that maintains working memory per tab, sees its prior decisions verbatim, and deliberates with extended thinking before acting.

## 2. Goals & Non-Goals

### Goals

1. The operator maintains a per-tab `OperatorMind` (goal, belief, open questions, tried-and-failed, next intent, last 5 turns) updated by the model itself each turn.
2. The mind is persisted to SQLite per `session_id`, surviving app restarts.
3. Each turn uses Claude extended thinking (budget 2000 tokens default) to reason before emitting both a `mind_update` and an `action` in a single structured response.
4. The operator panel renders the live mind state (belief, next intent, recent turns with collapsible thinking traces).
5. Closing a tab with non-empty mind triggers a confirmation modal.
6. Behavior is gated by `operator.mind_v2` setting (default `false` at merge); old code path remains intact for instant rollback.
7. Subjective: after one week of dogfood, the operator feels meaningfully more intelligent — coherent across turns, doesn't repeat failures, "remembers" yesterday's work.

### Non-Goals (out of scope, deferred)

- **Tools** (`read_file`, `git_status`, `tab_peek`, `recall_search`) — deferred to OP-MIND-2.
- **Multi-step ReAct loops** (think → tool → think → act) — deferred to OP-MIND-2.
- **Cross-tab correlation** (super-agent reconciling minds across tabs) — separate layer, not this spec.
- **Vector embeddings over historical minds** — possible OP-MIND-3.
- **Sharing minds across sessions of the same project (familiars)** — YAGNI.
- **User-editable mind in the UI** — possible follow-up if dogfood demands it.

## 3. Architecture

### 3.1 Modules

| Path | Role |
|---|---|
| `crates/app/src/operator_mind.rs` | **New.** Pure logic: types, `apply()`, caps, prompt-block render, response parser. No PTY, no HTTP, no DB. |
| `crates/app/src/operator.rs` | Hooks: `OperatorState` gains `Option<OperatorMind>`; turn-loop uses it when `mind_v2` is on. Old path untouched when off. |
| `crates/app/src/storage.rs` | New table `operator_mind`; `mind_load` / `mind_save` / `mind_delete`. |
| `crates/app/src/settings.rs` | `operator.mind_v2: bool` (default `false`), `operator.mind_thinking_budget: u32` (default `2000`, cap `4000`). |
| `ui/src/operator/` | Panel renders belief, next intent, recent turns; `MindLossModal` on tab close. |

### 3.2 Per-turn flow (when `mind_v2 == true`)

```
operator turn tick
  ↓
load tail + idle (unchanged)
  ↓
mind = state.mind  (already hydrated on session open)
  ↓
render_user_message:
    <mind>...</mind>                  ← mutable, no cache breakpoint
    <recent-decisions>...</recent>    ← mutable
    <tail>...</tail>                  ← mutable
  ↓
system blocks (persona, mission, mind-v2 directive) — all cache_control: ephemeral
  ↓
anthropic call with thinking { enabled, budget_tokens: 2000 }
  ↓
parse response → { thinking_blocks, mind_update, action }
  ↓
mind.apply(mind_update); mind.record_turn(TurnRecord { saw, thought, action, executed })
  ↓
state.mind_dirty = true   (debounced flusher saves to SQLite)
  ↓
execute action (existing safety pipeline + the \r auto-submit fix)
  ↓
emit OperatorPanelEvent::MindUpdated to UI
```

### 3.3 Why a separate module, no new crate

Pure logic (parse, merge, render) is testable without Tauri, PTY, or HTTP. Interaction code (Anthropic call, DB I/O) stays in `operator.rs` where existing call sites live. A new crate would add workspace overhead with no current testability win; revisit if `operator_mind.rs` grows past ~800 LOC.

## 4. Data Model

### 4.1 `OperatorMind`

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct OperatorMind {
    pub goal: String,                              // seed: mission title
    pub belief: String,                            // 1–3 sentences, updated every turn
    pub open_questions: Vec<String>,               // cap 5
    pub tried_failed: Vec<String>,                 // cap 5
    pub next_intent: String,                       // coherence check next turn
    pub recent: VecDeque<TurnRecord>,              // cap 5
    pub turn_count: u64,
    pub updated_at: DateTime<Utc>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TurnRecord {
    pub turn: u64,
    pub at: DateTime<Utc>,
    pub saw: String,                               // ≤ 400 chars, ANSI-stripped
    pub thought: String,                           // ≤ 200 chars, summary of thinking
    pub action: TurnAction,
    pub executed: bool,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum TurnAction {
    Reply { text: String },
    Execute { command: String },
    Escalate { notification: String },
    Ignore,
}

#[derive(Debug, Clone, Deserialize, Default)]
pub struct MindUpdate {
    pub goal: Option<String>,
    pub belief: Option<String>,
    pub open_questions_set: Option<Vec<String>>,   // full replace
    pub tried_failed_append: Option<Vec<String>>,  // append + cap-FIFO
    pub next_intent: Option<String>,
}
```

### 4.2 Caps and merge semantics

- `apply()` enforces caps server-side. If the model tries to push a 6th item into `tried_failed`, the oldest is dropped (FIFO).
- `open_questions_set` is **replace** semantics; the model resends the full list each time it changes. To clear the list, the model sends `Some(vec![])`. To leave unchanged, it omits the field.
- `tried_failed_append` is **append** semantics; the model only sends new entries, server caps the total.
- The system prompt tells the model about caps so it prioritizes correctly.

### 4.3 Serialization sizes (estimated)

| Field | Typical | Worst (at cap) |
|---|---|---|
| goal | 50 chars | 200 |
| belief | 200 chars | 600 |
| open_questions | 5 × 80 chars | 5 × 200 |
| tried_failed | 5 × 80 chars | 5 × 200 |
| next_intent | 80 chars | 300 |
| recent (5 turns) | 5 × 600 chars | 5 × 1200 |

Total prompt-block worst case: ~9 KB. Typical: ~3 KB. Negligible vs 16 KB tail.

## 5. Persistence (SQLite)

### 5.1 Schema

```sql
CREATE TABLE IF NOT EXISTS operator_mind (
    session_id  TEXT PRIMARY KEY,
    json        TEXT NOT NULL,
    turn_count  INTEGER NOT NULL,
    updated_at  TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_operator_mind_updated_at
    ON operator_mind(updated_at);
```

JSON blob (not normalized) because the mind is an aggregate, only ever read/written whole, never queried by subfield.

### 5.2 Storage API

```rust
async fn mind_load(&self, session_id: SessionId) -> Result<Option<OperatorMind>>;
async fn mind_save(&self, session_id: SessionId, mind: &OperatorMind) -> Result<()>;
async fn mind_delete(&self, session_id: SessionId) -> Result<()>;
```

### 5.3 Save policy

- Every mind mutation sets `state.mind_dirty = true`.
- A per-session `tokio::interval(500ms)` flushes when dirty. No-op when not dirty.
- `SessionClosed` and app-exit do a final synchronous flush (timeout 2s).
- Save failures log warning, keep `mind_dirty = true`, retry next tick. Never block the turn-loop.

### 5.4 Load policy

- On `SessionOpened` for a previously-persisted tab: `mind_load(session_id)`. `Some` → hydrate. `None` or error → start with `default()` and seed `goal` from mission title (empty string if no mission attached yet).
- New tabs always start with `default()` + best-effort seed.
- First-turn semantics: with `turn_count == 0`, the user message renders an empty `<mind>` block (only `goal` if seeded). The model's first `mind_update` is expected to populate `belief` and `next_intent`.

### 5.5 Staleness

- Reopening a tab with `now() - updated_at > 24h` does not invalidate the mind. The next user message gets a `<mind-stale hours_since_update="32" />` attribute. The model sees it and reverifies before acting.

### 5.6 Schema evolution

- `serde_json::from_str` failure → log error, `mind_delete`, return `None`. Mind reconstructs in 1 turn.
- No version column for v1; if a future change demands it, add a `version` field inside the JSON itself.

### 5.7 Garbage collection

- On confirmed tab delete, `mind_delete(session_id)` runs in the same transaction as tab deletion.
- Daily startup GC: `DELETE FROM operator_mind WHERE session_id NOT IN (SELECT id FROM tabs)`.

### 5.8 Confirm-on-delete UX

When the user requests tab close/delete:
- If `mind.turn_count == 0` → close directly, no modal.
- If `mind.turn_count > 0` → modal:
  - Title: **"Borrar tab y su memoria del operador?"**
  - Body shows `turn_count`, `updated_at` (relative), `goal`, and `belief` (truncated to 200 chars).
  - Buttons: `Cancelar` (default focus) | `Borrar de todas formas` (destructive style).
- `Cmd+W` triggers the modal flow when mind is non-empty.
- No "Export mind" in v1.

`tab_close` Tauri command splits into:
- `tab_close_check(tab_id) -> Option<MindPreview>`
- `tab_close_confirm(tab_id)`

Frontend calls `tab_close_check` first; if `Some(preview)`, opens `<MindLossModal preview={...} onConfirm={tab_close_confirm} />`; else closes directly.

## 6. Prompt Structure & Model Call

### 6.1 System prompt addition

A new static block appended to the system prompt when `mind_v2` is on:

```
# OPERATOR MIND (v2 protocol)

You maintain a persistent working memory for THIS tab across turns.
Each turn you receive your current mind state plus the latest tail.
You MUST emit, alongside your action:

1. A `mind_update` JSON object with ANY of these fields you want to change.
   Omit fields you don't want to change. Caps enforced server-side
   (oldest dropped if exceeded): open_questions ≤ 5, tried_failed ≤ 5.

   - goal (string): high-level objective in this tab. Set once, change rarely.
   - belief (string): your current 1–3 sentence understanding. UPDATE EVERY TURN if anything changed.
   - open_questions_set (string[]): full replace.
   - tried_failed_append (string[]): things that didn't work, with why.
   - next_intent (string): what you plan to do NEXT turn if conditions hold.

2. Your action (Reply / Execute / Escalate / Ignore), same schema as before.

CONTRACT:
- If your `next_intent` from last turn doesn't match what you're doing now,
  briefly explain in `belief` why you changed course.
- If something is in `tried_failed`, do NOT propose it again unless conditions
  clearly changed (and say what changed in `belief`).
- Use extended thinking to reason through ambiguity. Final `mind_update` and
  `action` should be the *result* of thinking, not the thinking itself.
```

All system blocks (persona, mission, mind-v2 directive) get `cache_control: ephemeral`.

### 6.2 User message structure

```
<mind turn="{n}" updated="{relative}" {stale-hours="32"}>
  <goal>{goal}</goal>
  <belief>{belief}</belief>
  <open-questions>
    {q_1}
    ...
  </open-questions>
  <tried-failed>
    {tf_1}
    ...
  </tried-failed>
  <next-intent>{next_intent}</next-intent>
</mind>

<recent-decisions>
  <turn n="{T-4}" at="...">
    saw: {saw}
    thought: {thought}
    action: {kind} {summary}
    {executed|blocked}
  </turn>
  ...up to 5...
</recent-decisions>

<tail kind="executor-output" cwd="{cwd}" idle_ms="{idle}">
{tail_excerpt}
</tail>
```

### 6.3 Anthropic API call

```rust
let req = MessagesRequest {
    model: "claude-opus-4-7",
    max_tokens: 4096,
    thinking: Some(ThinkingConfig::Enabled {
        budget_tokens: settings.operator.mind_thinking_budget,
    }),
    system: vec![
        SystemBlock { text: persona_block,        cache_control: Some(Ephemeral) },
        SystemBlock { text: mission_block,        cache_control: Some(Ephemeral) },
        SystemBlock { text: mind_v2_directive,    cache_control: Some(Ephemeral) },
    ],
    messages: vec![
        Message::user(vec![ContentBlock::text(mind_block + recent_block + tail_block)]),
    ],
    tools: None,
    tool_choice: None,
};
```

### 6.4 Output schema

The model returns thinking blocks plus a single text block containing JSON:

```json
{
  "mind_update": {
    "belief": "Executor finished Task 4, asking yes/no for file overwrite. Mtime conflict means user edited mission externally.",
    "next_intent": "Reply 'yes' if mtime delta < 5 min; escalate otherwise."
  },
  "action": {
    "kind": "Reply",
    "text": "yes",
    "rationale": "User saved mission 30s ago; conflict is benign."
  }
}
```

### 6.5 Thinking trace capture

- Full thinking blocks → `tracing::debug!` for audit.
- First ≤200 chars → `TurnRecord.thought` (re-enters next prompt).
- Full blocks → emitted to UI as `OperatorPanelEvent::ThinkingTrace { turn_id, blocks }` for collapsible display.

### 6.6 Latency & cost expectations

- Thinking budget 2000 tokens ≈ +1–2s per turn.
- Polling baseline relaxes from ~1s to ~3s (mind reduces need for high-frequency probing).
- System prompt cache hit ratio stays high (static per session).
- User message cache misses are expected (mind mutates) but message is small (~3 KB typical).
- Estimated net cost per turn: similar or slightly lower than current, due to relaxed spacing.

## 7. Errors, Edge Cases, Safety

### 7.1 Parse failures

- Invalid JSON → `tracing::warn!`, mind unchanged this turn, action downgrades to `IGNORE`.
- `state.consecutive_parse_failures += 1`. At 3 → emit `OperatorPanelEvent::Degraded` with suggestion to disable `mind_v2` for this tab. Do not auto-disable.

### 7.2 Thinking budget truncation

- `stop_reason == "max_tokens"` treated like parse fail (no commit, IGNORE).
- Two consecutive truncations → bump session-local budget by +1000 (cap 4000), in-memory only.

### 7.3 Save / load failures

- `mind_save` failure → log, retry next tick.
- `mind_load` corruption → log, `mind_delete`, default. Reconstructs in 1 turn.
- App-exit flush has 2s timeout; non-blocking.

### 7.4 Cap enforcement

- `apply()` enforces caps via FIFO. Model is not notified of drops; relies on `belief` to reconstruct what's important.

### 7.5 Loop detection upgrade

- Existing `decision_hash` check unchanged.
- **New guard:** compute a normalized "action signature" — for `Reply`, lowercased trimmed text; for `Execute`, the first token of the command. If the current turn's action signature appears (case-insensitive substring match) in any entry of `tried_failed`, force `Escalate` with notification `"operator about to repeat a known-failed action: <signature>"`. The `next_intent` continuity is informational (logged) but not part of the gate — the gate is signature-vs-tried_failed alone, to keep it simple and bypass-resistant.

### 7.6 Safety blocklist

- Action passes through `safety::is_dangerous` unchanged. Block → downgrade to Escalate. Mind records `tried_failed_append: ["attempted REPLY '...' — blocked by safety: <reason>"]`.

### 7.7 AOM toggle mid-session

- Mind survives AOM on/off. AOM only gates whether actions inject live; mind is independent context.

### 7.8 Mission change mid-session

- mtime change on mission file → next user message gets `<mission-changed previous_goal="..." />` marker. Model updates `goal` in `mind_update`.

### 7.9 Privacy / secrets in mind

- Model-authored text in `belief` and `tried_failed` could include tokens scraped from tail.
- Mitigation: existing safety masker runs over the mind **before persistence to SQLite**, not just before LLM upload. Pattern match → mask + log warning.

### 7.10 Feature flag off

- `mind_v2 == false` → no new code runs. Old path is bit-identical (validated by snapshot test of system prompt vs baseline).

## 10. Success Criteria

1. **Functional:** open tab, run 5 turns, close, reopen → belief and 5 recent_turns hydrate from SQLite.
2. **Coherence:** model never repeats a `tried_failed` action without justifying in `belief`.
3. **Latency:** p50 turn < 3s, p95 < 5s end-to-end with thinking budget 2000.
4. **Stability:** parse-failure rate < 2% over 1 week of real use.
5. **Subjective:** dogfood week 1 → operator feels coherent and aware. If not, default stays off and we open a post-mortem.

## 11. Open Risks

- **Cache thrash.** Mind blocks change every turn, breaking message-level cache. Mitigation: keep mutable blocks at the **end** of the user message; system prompt (largest, most expensive) stays fully cached.
- **Model misuses `tried_failed`.** It might over-record minor frictions and box itself in. Mitigation: directive emphasizes "things that *didn't work*, not minor adjustments"; loop guard only fires on exact-signature match.
- **Persistence-induced bugs surviving across restarts.** A bad belief from yesterday could mislead today. Mitigation: stale marker after 24h; `mind_load` failure → fresh default; if dogfood shows persistent confusion, add a "reset mind" button to the panel in a follow-up.
- **Thinking-budget cost.** 2000 tokens × Opus pricing × N tabs × N turns/min could add up. Mitigation: relaxed polling; if cost-per-hour spikes >2× baseline, ship a lower-budget default (1000) and revisit.
