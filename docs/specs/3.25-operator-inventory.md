# 3.25 — Operator Inventory

## Goal
Develop an "Operator Inventory" system that enables operators to dynamically store, recall, and consult relevant skills (e.g., `Skill` type) and documentation in real-time within the terminal, while collaborating with the agent on task execution.

## Out of scope
- Integration with external APIs for skill acquisition — this task focuses only on managing inventory within the terminal environment.
- Advanced AI-driven suggestions — while the system will store and retrieve skills and documentation, it will not generate suggestions autonomously.
- Persistent storage across sessions — skills and documentation will be session-specific unless specified in future iterations.
- UI/UX overhaul — this spec does not cover detailed frontend designs or visual improvements.
- Modifications to existing skill types in `Kind` — no alteration to existing `Skill` definitions, only extensions where necessary.

## Acceptance criteria
- [ ] Operators can add skills or documentation to the inventory dynamically during a session.
- [ ] Operators can recall items from the inventory to assist in task execution or decision-making.
- [ ] The inventory is linked to the `Skill` type defined in `model.rs` and correctly serializes/deserializes data.
- [ ] No existing agent capabilities (e.g., `Tool` or `Kind`) are disrupted by the introduction of the inventory system.
- [ ] Error handling is implemented to capture invalid or malformed entries added to the inventory.

## File boundaries
- **Create**:
  - `crates/capabilities/src/operator_inventory.rs`: To house the logic for managing inventory, including adding, recalling, and error handling.
- **Touch**:
  - `crates/capabilities/src/model.rs`: To extend or align the `Skill` type with the inventory logic.
  - `crates/capabilities/src/lib.rs`: To integrate the `operator_inventory` module into the capabilities.
- **DO NOT touch**:
  - `crates/agent/src/`: This module handles agent-specific logic and does not interact directly with the inventory in this initial implementation.

## Complexity
**Medium** — This task involves introducing a new module, defining dynamic inventory management logic, connecting it to existing structures (`Skill`), ensuring serialization/deserialization compatibility, and providing error handling. It may take 2–3 development sessions.

## Open questions
- Should the inventory persist across sessions, or strictly remain session-specific in this implementation?
- Should the "Operator Inventory" support any metadata (timestamps, categories) for stored items, or focus solely on minimal structures like `Skill`?
- Will operators interact with the inventory through the terminal interface or an additional UI within the agent panel?
