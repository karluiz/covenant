# 3.10 — Mission Drafts

## Goal

Permitir escribir specs desde cero dentro de Covenant con un wizard guiado que respeta el formato de `_template.md`, manteniéndolos como **Drafts** hasta que el usuario decida publicarlos como missions ejecutables por AOM.

Hoy "Set Mission" solo permite seleccionar archivos `.md` ya existentes; un usuario que tiene una idea nueva debe abrir el editor externo, copiar el template manualmente y crear el archivo. Este spec elimina esa fricción y formaliza el estado intermedio "borrador".

## Out of scope

- Colaboración multi-user sobre el mismo draft (single-user local).
- Historial / versionado dentro de la app — git ya lo cubre.
- Templates alternativos al `_template.md` (un único formato canónico).
- Ejecución directa de drafts por AOM — un draft debe publicarse primero.
- Asistencia LLM para `Goal`, `File boundaries` o `Complexity` (riesgo de alucinación sin contexto del repo; postergado a v2).
- Edición rica WYSIWYG; el editor es CodeMirror reusando el de Structure.
- Sincronización con servicios externos (Linear, GitHub Issues, etc.).

## Acceptance criteria

- [ ] Sidebar muestra una entrada **Drafts** (icon-only, Lucide `file-pen`, tooltip "Drafts") que abre un panel fullscreen estilo Docs hub.
- [ ] El panel lista los archivos `docs/specs/drafts/*.md` con title + `updated_at` + botón `+ New draft`; click abre el wizard.
- [ ] El wizard presenta 6 secciones (Goal, Out of scope, Acceptance criteria, File boundaries, Complexity, Open questions) en una página scrollable; cada una con header, tooltip de ayuda y editor CodeMirror; Complexity es segmented control `small | medium | large`.
- [ ] El draft se guarda automáticamente en `docs/specs/drafts/<slug>.md` con frontmatter YAML (`status: draft`, `title`, `slug`, `created_at`, `updated_at`, `llm_calls`) en cada blur del editor y cada 30s.
- [ ] Las secciones `Out of scope`, `Acceptance criteria` y `Open questions` muestran un botón **✨ Help** que llama a Haiku 4.5 vía `agent::dispatch()`, devuelve 3 sugerencias en un popover, y permite insertarlas (append) en el textarea; cap configurable (default 20 calls/draft) que deshabilita el botón al alcanzarse.
- [ ] El botón **Publish** está deshabilitado hasta que Goal + Acceptance criteria + Complexity tengan contenido no-placeholder; al click abre un modal con ID auto-sugerido (max ID en `docs/specs/` + 0.1) y slug derivado del title, ambos editables y validados como únicos.
- [ ] Confirmar la publicación mueve el archivo a `docs/specs/<id>-<slug>.md`, elimina el frontmatter, reescribe el `# Draft — X` a `# <id> — X`, y muestra un toast con opción "Open in Set Mission".
- [ ] Drafts pueden eliminarse desde la lista mediante confirm modal (sin papelera; confiamos en git).
- [ ] Tests Rust cubren `list_drafts`, `save_draft`, `publish_draft` (colisión de slug/ID, validación de frontmatter, atomicidad del move); tests TS cubren transiciones del wizard, debounce de autosave y lógica de completitud.

## File boundaries

- **Create**:
  - `crates/app/src/drafts.rs` (≤ 250 líneas) — comandos Tauri `list_drafts`, `read_draft`, `save_draft`, `publish_draft`, `delete_draft`; parsing de frontmatter; validación.
  - `ui/src/drafts/panel.ts` (≤ 300 líneas) — vista lista + wizard fullscreen.
  - `ui/src/drafts/wizard.ts` (≤ 350 líneas) — 6 pasos, autosave, ✨ Help popover, publish modal.
  - `ui/src/drafts/api.ts` (≤ 80 líneas) — typed wrappers de los 5 comandos.
  - `docs/specs/drafts/.gitkeep` — placeholder para que el directorio exista.
- **Touch**:
  - `ui/src/main.ts` (≤ 15 líneas) — registrar entrada Drafts en sidebar y router.
  - `ui/src/api.ts` (≤ 30 líneas) — re-export de los wrappers.
  - `crates/app/src/lib.rs` (≤ 10 líneas) — registrar comandos con `tauri::generate_handler!`.
  - `ui/src/styles.css` (≤ 100 líneas) — estilos del panel, wizard, popover ✨ y publish modal.
- **DO NOT touch**:
  - `crates/agent/` — la asistencia LLM va vía `agent::dispatch()` ya existente; no agregar lógica nueva ahí.
  - `ui/src/tabs/` — los drafts no son tabs, viven en su propio panel.
  - `ui/src/settings/` — el toggle del cap LLM se posterga (v2); por ahora hardcoded a 20.
  - `docs/specs/_template.md` — formato canónico, no modificar.
  - Specs publicados existentes (`3.1-*` … `3.9-*`) — no migrar ni reformatear.

## Complexity

`medium` (2–3 sesiones AOM).

- Sesión 1: backend Rust (drafts.rs + tests + comandos registrados).
- Sesión 2: frontend lista + wizard + autosave + publish.
- Sesión 3: ✨ Help integration + cap + pulido CSS + e2e manual.

## Open questions

- ¿El cap de 20 calls/draft debe ser global o configurable en Settings desde v1? (Recomendación: hardcoded v1, settings v2.)
- ¿Validar que el ID sugerido respete el major version actual (3.x) o permitir saltar a 4.x desde la UI? (Recomendación: permitir override libre, sin enforcement.)
- ¿El toast post-publish con "Open in Set Mission" debe seleccionar el spec en la tab activa o pedir tab destino si hay varias? (Recomendación: tab activa, consistente con el flow actual de Set Mission.)
