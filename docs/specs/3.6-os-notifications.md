# 3.6 — OS Notifications

## Goal

Send native macOS notifications when the master operator or AOM
needs the user's attention — so the user can step away from the
machine and trust they will be paged on a real blocker, not when
everything is fine.

## Out of scope

- **Cross-platform support** — macOS only for v1. Windows / Linux
  notification routing is its own spec.
- **Custom notification UI** — use `tauri-plugin-notification`
  defaults. No custom layout, no rich content.
- **Sound design** — OS default sound. No bundled audio assets.
- **Notification history view inside Covenant** — that's what
  `⌘⇧R` (morning report) and AFK mode (3.4) are for.
- **Notifications for trivial events** — `BlockFinished` with
  `exit_code == 0`, recall searches, settings changes, etc. No.
- **Per-tab notification opt-in/out** — global toggles per
  trigger type only. Per-tab is overkill at v1.
- **Re-arming dismissed notifications** — a dismissed notification
  is gone; we do not re-fire.
- **In-app banner mirror** — the toast system is separate; this
  spec only adds OS-level pings.

## Acceptance criteria

- [ ] On first relevant trigger after install, Tauri requests
      macOS notification permission.
- [ ] OS notification fires when:
      (1) the operator emits an `ESCALATE` action (blocked,
      awaiting user) — title "Operator paused", body shows
      one-line reason;
      (2) an AOM session hits an unrecoverable error (e.g., budget
      blown, repeated safety blocks);
      (3) an AOM session completes — title "AOM finished",
      configurable per setting (default: on).
- [ ] Clicking the notification focuses the Covenant window AND
      switches to the originating tab.
- [ ] No notification fires when the Covenant window is currently
      focused (configurable; default: suppress).
- [ ] Settings page exposes three toggles (one per trigger type)
      and one global "suppress when window focused".
- [ ] Throttle: at most one notification per trigger type per 30
      seconds across all tabs. The throttled-out events still log
      via `tracing`, just no popup.
- [ ] `cargo check -p covenant` passes; new tests cover the
      throttle logic and trigger gating.

## File boundaries

- **Create**:
  - `crates/app/src/notify.rs` — `Notifier` struct with throttle,
    settings-aware emit, click-routing payload (≤ 180 lines).
- **Touch**:
  - `crates/app/src/operator.rs` — emit on `ESCALATE`. Max 20
    lines added; do NOT alter decision logic.
  - `crates/app/src/aom.rs` — emit on terminal events
    (error / complete). Max 25 lines added.
  - `crates/app/src/lib.rs` — wire `Notifier`, register new
    commands if any. Max 30 lines.
  - `crates/app/src/settings.rs` — add four boolean fields with
    serde defaults. Max 40 lines.
  - `crates/app/Cargo.toml` — add `tauri-plugin-notification`
    dependency.
  - `crates/app/tauri.conf.json` — register the plugin under
    `app.security.csp` if needed AND under `plugins`.
  - `ui/src/settings/panel.ts` — render the four toggles. Max 60
    lines added.
  - `ui/src/api.ts` — only if a settings read/write wrapper is
    missing for the new fields. Max 20 lines.
- **DO NOT touch**:
  - `crates/agent/`, `crates/session/`, `crates/blocks/`,
    `crates/pty/` — notifications are an app-level concern.
  - `ui/src/blocks/`, `ui/src/recall/`, `ui/src/operator/`,
    `ui/src/aom/`, `ui/src/structure/`, `ui/src/tabs/` — UI is
    settings-only.

## Complexity

`small`-`medium` — 1 AOM session if the plugin is already in deps,
2 sessions if it needs to be added and signed.

## Open questions

- **Click-to-focus**: macOS may not surface a notification's
  click on a Tauri window unless the app handles it via
  `tauri::WindowEvent`. Verify the plugin's docs at run time;
  ESCALATE if click-routing requires custom Swift code.
- **Focus modes**: should we respect macOS Focus / Do Not Disturb?
  The OS already handles suppression — we should NOT try to query
  Focus state and silently swallow events.
- **Test coverage for the OS layer**: the throttle and trigger
  gating are unit-testable; the actual `notify_send` call is
  not. Acceptance only requires logic-level coverage.
