# 3.17 — Spec pending recovery: badge + contextual link

## Goal

Cuando 3.16 detecta un spec nuevo y emite `spec:candidate`, hoy el toast
dura ~30 s y si el coordinador lo pierde el evento se evapora: tiene que
recordar el path, abrirlo a mano y asignarlo via mission picker. Esta
spec agrega dos redes de seguridad sobre 3.16, sin tocar la detección:

1. **Badge persistente en el tab** mientras haya candidates pending para
   esa tab. Un click abre un popover con la lista de specs pendientes y
   acciones rápidas (`Asignar / Descartar / Abrir`).
2. **Cmd+click contextual sobre paths de specs** en el output del agente
   (y en cualquier render de markdown dentro de la app): si el path
   matchea `docs/specs/**/*.md`, en vez del comportamiento default
   (revelar en Finder / abrir en editor) se muestra un mini-menú con
   `Abrir / Asignar a esta sesión / Asignar a otra sesión / Revelar en
   Finder`.

El objetivo es que el "wow moment" de 3.16 sobreviva la atención
distraída del coordinador.

Además, esta spec corrige un **bug observado en 3.16**: el toast usa
`setMissionPathForActiveTab(path)`, que resuelve "active" al momento
del click. Como el toast se renderiza en múltiples tabs elegibles y
puede sobrevivir cambios de tab, hacer click en `Set as mission` desde
un toast "viejo" termina asignando el spec a la tab activa actual, no
a la tab donde apareció el toast. Hay que **anclar el target tab al
momento de emitir el toast**, no al momento del click.

## Out of scope

- **Cambios al detector de 3.16** — esta spec sólo *consume*
  `spec:candidate` y los pending state que ya existen en
  `spec-prompt.ts`. No tocamos `crates/app/src/spec_detector.rs`.
- **Cambios al toast de 3.16** — el toast sigue apareciendo igual; el
  badge es complementario, no reemplazo.
- **Detección de paths fuera de `docs/specs/**/*.md`** — el menú
  contextual NO se activa para READMEs, drafts crudos, otros `.md` del
  repo, ni paths externos. Mantener predecibilidad sobre cobertura.
- **Asignar specs como referencia/contexto** — "asignar" sigue
  significando exactamente `setMissionPathForActiveTab(path)`. No
  agregamos un concepto nuevo de "spec attached pero no mission".
- **Multi-tab batch UI** — un badge por tab; si hay >1 candidate, el
  badge muestra contador y el popover lista. No agregamos vista global
  cross-tab (eso es 3.18 si llega).
- **Persistir dismiss en SQLite** — el dismiss-state sigue siendo
  in-memory como 3.16. Restart de app limpia el badge.
- **Editar la mission asignada via popover** — popover sólo asigna; si
  ya hay mission, el flujo de cambiar mission es el existente (mission
  picker / editable modal).
- **Settings para deshabilitar el badge o el menú contextual** — v1
  hardcoded ON. Configurable post-feedback.

## Acceptance criteria

### Badge en tab

- [ ] Cuando `getPendingSpecCandidate(tabId)` (de 3.16) retorna ≥1
      candidate no-dismissed para una tab, el tab renderiza un badge
      `📎` (o icono Lucide `paperclip`) al lado del nombre, con
      contador si hay >1 (`📎2`).
- [ ] El badge persiste hasta que: (a) se asigna un candidate (queda
      sólo si hay otros pending), (b) se descartan todos, o (c) restart
      de app.
- [ ] Click en el badge → popover anclado al tab que muestra:
      - Lista de candidates pending (path relativo + Goal snippet de
        3.16, ≤200 chars).
      - Por cada item: botones `[Asignar] [Abrir] [Descartar]`.
- [ ] Click `[Asignar]` → `setMissionPathForActiveTab(path)`, candidate
      removido de pending, popover cierra si era el último.
- [ ] Click `[Abrir]` → abre el `.md` en el viewer/editor existente
      (mismo handler que cmd+click default sobre `.md`).
- [ ] Click `[Descartar]` → marca dismissed in-memory (mismo store que
      3.16), candidate removido del pending list de esa tab.
- [ ] Click fuera del popover → cierra sin acción.
- [ ] Tab sin candidates pending NO muestra badge (no placeholder).
- [ ] El badge respeta el filtro per-tab de 3.16: si la tab no tiene
      operator asignado o su `cwd` no está bajo el `repo_root` del
      spec, el candidate no aparece en su badge.
- [ ] Aceptar el toast de 3.16 también limpia el badge (mismo state).
- [ ] Asignar via mission picker manual NO limpia el badge
      automáticamente — el candidate sigue pending hasta dismiss
      explícito o restart. *(Razón: el picker no sabe que ese path
      provino de un candidate; agregar el binding agrega acoplamiento
      no necesario.)* **Reconsiderar si el feedback dice que es
      confuso.**

### Fix bug 3.16: toast asigna a la tab equivocada

- [ ] El toast de 3.16 (`spec-prompt.ts`) captura el `tabId` destino
      en el momento de **renderizar** el toast, no en el momento del
      click. Click `[Set as mission]` → `setMissionPathForTab(capturedTabId, path)`,
      no `setMissionPathForActiveTab(path)`.
- [ ] Si la tab capturada se cerró entre render y click → fallback a
      la tab activa actual con un confirm modal
      (`Tab original cerrada — ¿asignar a la tab activa?`). Nunca
      asignar silenciosamente a una tab inesperada.
- [ ] Mismo fix aplica al popover del badge (3.17): cada item del
      popover lleva el `tabId` del badge que lo abrió.
- [ ] Test TS: `setMissionFromToast_usesCapturedTab` — simula cambio
      de tab activa entre render y click, verifica que asigna a la
      capturada.

### Cmd+click contextual sobre spec paths

- [ ] En cualquier render de output del agente que contenga un link a
      un path matcheando `docs/specs/**/*.md` (excluyendo
      `_template.md` y `drafts/**`, igual que 3.16), cmd+click NO
      ejecuta el handler default. En su lugar, muestra un mini-menú
      contextual anclado al cursor con:
      - `Abrir spec` (default, equivalente al cmd+click viejo)
      - `Asignar a esta sesión`
      - `Asignar a otra sesión…` (sub-menu o modal con lista de tabs)
      - `Revelar en Finder`
- [ ] `Abrir spec` → abre `.md` en viewer/editor (mismo handler que
      cmd+click previo a esta feature).
- [ ] `Asignar a esta sesión` → `setMissionPathForActiveTab(path)`. Si
      la tab activa ya tiene mission, confirmar con modal de overwrite
      (reusa el modal de 3.10/3.11 si existe; si no, modal simple
      `[Reemplazar] [Cancelar]`).
- [ ] `Asignar a otra sesión…` → modal con lista de tabs abiertas
      (nombre + cwd + mission actual si tiene). Click en una tab →
      `setMissionPathForTab(tabId, path)`. Si la tab destino tiene
      mission, mismo modal de overwrite.
- [ ] `Revelar en Finder` → comportamiento existente (`revealInFinder`).
- [ ] Paths que NO matchean `docs/specs/**/*.md` → cmd+click conserva
      su comportamiento actual sin cambios.
- [ ] Tests unitarios para el matcher de paths
      (`isSpecPath(path): boolean`): cubre matches válidos, rechaza
      `_template.md`, `drafts/`, paths fuera de `docs/specs/`, paths
      relativos vs absolutos.
- [ ] `npx tsc --noEmit` pasa.

## Complexity

`small` (1 sesión AOM).

- Frontend puro. Backend de 3.16 ya provee todo lo necesario.
- Badge + popover ~ 180 líneas; menú contextual + matcher ~ 150 líneas;
  tests ~ 160 líneas.
- Riesgos:
  - **Hook de cmd+click sobre output del agente**: depende de cómo se
    renderiza el output (HTML del LLM con links, o terminal output con
    OSC 8 hyperlinks). Si los paths llegan como texto plano sin
    estructura, hay que parsearlos a posteriori — ESCALAR si requiere
    tocar el render del agente más allá de un listener delegado.
  - **`onPendingChange` en `spec-prompt.ts`**: el store actual puede ser
    síncrono y no exponer eventos. Agregar un `EventTarget` simple es
    OK; si requiere refactor del state model → ESCALAR.
  - Modal de tab-picker: si no hay un componente de modal genérico
    reusable, escribirlo desde cero suma ~50 líneas — aceptable, pero
    notar en el plan.

## Open questions

- ¿El badge debe ser un icono distinto al `📎`? Lucide `paperclip` es
  sobrio y consistente con la estética. Recomendación: `paperclip` con
  color `--accent` cuando hay pending.
- ¿"Asignar a otra sesión" debe limitar a tabs cuyo `cwd` esté bajo el
  `repo_root` del spec (mismo filtro que 3.16)? Recomendación: SÍ, es
  consistente y evita asignar specs a tabs que no pueden trabajarlos.
- ¿Asignar via mission picker manual debería limpiar el badge? Hoy NO
  (ver acceptance). Si feedback dice que es confuso, agregar un hook
  en `setMissionPathForActiveTab` que matchee path contra pending y
  limpie automáticamente. Cambio de 1 línea, post-v1.
- ¿El menú contextual debe aparecer también en cmd+click sobre
  `docs/specs/**` desde el structure file tree (3.3)? v1: NO,
  mantenemos scope a output del agente. Si el coordinador lo pide,
  extender en v2.
