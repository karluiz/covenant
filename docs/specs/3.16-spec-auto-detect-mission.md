# 3.16 — Auto-detect spec → propose mission

## Goal

Cuando aparece un spec nuevo en el repo (publicado vía Drafts o creado por
la skill `superpowers:brainstorming`), Covenant lo detecta y propone
automáticamente "¿usar como misión?" en cada tab elegible — toast en vivo
durante el trabajo y modal de "última llamada" al pulsar Engage AOM en
una tab sin misión. El objetivo es eliminar la fricción de cargar
manualmente un spec recién escrito y dar un "wow moment" al coordinador
mostrándole que la app entiende el contexto que acaba de producir.

## Out of scope

- **Auto-asignar sin confirmación** — siempre se pregunta. El usuario es
  quien acepta la misión.
- **Atribución por sesión / agente** — no rastreamos quién creó el spec.
  El criterio es "archivo nuevo en el repo desde el snapshot inicial".
- **Drafts no-publicados como candidatos** — el contrato de 3.10/3.11
  (sólo specs publicados son misión válida) NO cambia. Drafts crudos no
  disparan toast.
- **Edits a specs ya vistos** — sólo archivos nuevos. Editar un spec
  existente no re-dispara nada (sería ruido constante mientras escribes).
- **Multi-spec batch UI** — un toast por spec; no agrupar "3 specs
  nuevos" en v1.
- **Watchers en otros directorios** (Linear, GitHub, paths arbitrarios).
- **Cambios al contrato de mission** (sigue siendo path → archivo `.md`
  publicado).
- **Cambios al motor de Operator / AOM** — esta feature SÓLO observa el
  filesystem y emite candidates; el `setMission` final reusa el camino
  existente.
- **Auto-engage AOM al detectar spec** — el toast propone misión, no
  arranca AOM. AOM se engagea como hoy (`⌘⇧A` o banner).

## Acceptance criteria

- [ ] Al primer arranque en un repo, un **snapshot** inserta todos los
      specs preexistentes en `seen_specs` con `first_seen_at = now`.
      No se emite ningún `spec:candidate` por specs preexistentes.
- [ ] Crear `docs/superpowers/specs/<YYYY-MM-DD>-<topic>-design.md` →
      detector emite `spec:candidate { source: Superpowers }`. Toast
      aparece en cada tab del repo que (a) no tiene misión, (b) tiene
      operator asignado.
- [ ] Crear `docs/specs/<id>-<name>.md` directamente (sin pasar por el
      wizard de Drafts) → mismo flujo que arriba con
      `source: Covenant`. (Red de seguridad — `_template.md` y
      `drafts/**` se ignoran.)
- [ ] Publicar un draft via wizard de Drafts (3.10/3.11) → el evento
      `mission:published` existente se reenvía al detector, que hace
      upsert en `seen_specs` y emite `spec:candidate { source: Covenant }`.
      El watcher de FS también lo verá pero el path ya estará en la
      tabla → ignorado (sin doble disparo).
- [ ] Editar un spec ya visto NO emite candidate.
- [ ] Toast (en statusbar, reusa estilos de 3.7) muestra:
      - Label según source: `Mission published: <file>` (Covenant) o
        `New spec detected: <file>` (Superpowers).
      - Snippet del Goal (≤ 200 chars) extraído del markdown.
      - Botones `[Set as mission] [Dismiss]`.
- [ ] Click `Set as mission` → llama `setMissionPathForActiveTab(path)`
      (existente). El toast cierra en esa tab. Otras tabs siguen viendo
      su toast (cada tab decide independiente).
- [ ] Click `Dismiss` o auto-timeout (30 s) → toast cierra. La tab marca
      el candidate como dismissed para esta sesión (memoria, no SQL):
      no vuelve a aparecer hasta el próximo restart de app.
- [ ] Tab con misión activa NO recibe toast (no interrumpir).
- [ ] Tab sin operator asignado NO recibe toast.
- [ ] Tab cuyo `cwd` no está dentro del `repo_root` del spec NO recibe
      toast.
- [ ] **Engage AOM guard**: al pulsar Engage AOM en una tab sin misión,
      si hay un candidate pending (no dismissed) emitido en los últimos
      10 min para esta tab → se muestra modal de confirmación
      "Detectamos `<file>` — ¿usar como misión antes de dormir?
      `[Use it] [Engage without mission] [Cancel]`". Si no hay
      candidate pending → flujo normal (engage sin misión).
- [ ] Engage AOM tras dismiss explícito NO muestra el modal.
- [ ] `cargo test -p covenant` cubre:
      - `snapshot_initial_skips_existing` — snapshot no emite events.
      - `new_file_emits_candidate` — archivo nuevo bajo
        `docs/superpowers/specs/` dispara event.
      - `published_event_dedupe` — `mission:published` + watcher hit
        sobre el mismo path emite UN solo event.
      - `template_and_drafts_ignored` — `_template.md` y `drafts/**`
        no disparan.
      - `extract_goal_snippet` — parser de Goal corta a 200 chars y
        strippea markdown básico.
- [ ] `npx tsc --noEmit` pasa. Tests TS para `specPromptState` cubren
      el filtro per-tab (cwd ⊂ repo_root, sin misión, operator
      assigned), dismiss persistence in-memory, y la lookup
      `getPendingSpecCandidate(tabId)` con expiración 10 min.

## File boundaries

- **Create**:
  - `crates/app/src/spec_detector.rs` (≤ 350 líneas) — watcher de FS
    via `notify`, snapshot inicial, dedupe contra `seen_specs`,
    clasificador `SpecSource::{Covenant, Superpowers}`, parser de
    Goal snippet, emisor del IPC event `spec:candidate`. Tauri
    commands: `subscribe_spec_candidates(repo_root)`,
    `mark_spec_seen(repo_root, path)`.
  - `ui/src/aom/spec-prompt.ts` (≤ 220 líneas) — listener del evento,
    filtro per-tab, render del toast en statusbar, dismiss state
    in-memory (Map<tabId, Set<path>>), cache de candidates pending
    para el guard de Engage AOM.
  - `ui/src/aom/spec-prompt.test.ts` (≤ 120 líneas).
- **Touch**:
  - `crates/app/Cargo.toml` (≤ 3 líneas) — añadir `notify = "6"` (o
    versión consistente con workspace si ya existe en otro crate;
    ESCALAR si requiere bump de workspace).
  - `crates/app/src/storage.rs` (≤ 25 líneas) — migración:
    `CREATE TABLE seen_specs(repo_root TEXT NOT NULL, path TEXT NOT
    NULL, first_seen_at INTEGER NOT NULL, PRIMARY KEY(repo_root,
    path))`.
  - `crates/app/src/drafts.rs` (≤ 30 líneas) — al emitir
    `mission:published`, también notificar al `SpecDetector` para
    upsert en `seen_specs` + emit `spec:candidate { source: Covenant }`.
  - `crates/app/src/lib.rs` (≤ 10 líneas) — registrar Tauri commands
    y arrancar el watcher al abrir un repo.
  - `ui/src/api.ts` (≤ 30 líneas) — wrappers
    `subscribeSpecCandidates`, `markSpecSeen`,
    `getPendingSpecCandidate`.
  - `ui/src/aom/banner.ts` (≤ 40 líneas) — interceptar Engage AOM
    cuando la tab no tiene misión: si
    `getPendingSpecCandidate(tabId)` retorna candidate → modal de
    confirmación; si no → flujo normal.
  - `ui/src/main.ts` (≤ 15 líneas) — bootstrap del listener
    (`spec-prompt.ts`) al abrir repo.
  - `ui/src/styles.css` (≤ 80 líneas) — estilos del toast en
    statusbar (reusa `--bg-overlay`, `--border`, `--accent`); estilos
    del modal "última llamada" (reusa modal base existente).
- **DO NOT touch**:
  - `crates/agent/`, `crates/blocks/`, `crates/session/`,
    `crates/pty/`.
  - `crates/app/src/operator.rs`, `aom.rs`, `safety.rs`,
    `mission_persistence.rs`, `cost.rs` — el contrato de
    Operator/AOM no cambia.
  - `ui/src/operator/`, `ui/src/recall/`, `ui/src/blocks/`,
    `ui/src/structure/`, `ui/src/settings/`, `ui/src/drafts/` (los
    Drafts publican; el detector escucha el evento existente, no se
    refactoriza el wizard).
  - `docs/specs/_template.md` y specs publicados existentes.

## Complexity

`medium` (2 sesiones AOM).

- Backend: watcher + tabla + parser + tests ~ 350 líneas.
- Frontend: toast + filtro + cache + guard de engage ~ 220 líneas.
- Riesgos:
  - **`notify` no está en `crates/app/Cargo.toml`**. Añadirlo es OK,
    pero si requiere bump del workspace o conflicta con una versión
    transitiva → ESCALAR antes de seguir.
  - Watcher de FS en macOS bajo Tauri (FSEvents). Si el watcher no
    detecta cambios fiablemente en el sandbox de la app → ESCALAR;
    fallback aceptable es polling cada 5 s del directorio (degradado
    pero funcional).
  - Detección de `repo_root` por tab: usar el `cwd` actual y subir
    hasta encontrar `.git/`. Si una tab no está en un repo, el
    detector no se monta para ella.

## Open questions

- ¿El snapshot inicial debe persistir un marcador "snapshot done" por
  repo, o re-correr en cada arranque es suficiente (idempotente porque
  es upsert)? Recomendación: re-correr siempre, simplifica.
- ¿Qué pasa si dos tabs en el mismo repo aceptan el toast del mismo
  spec? Ambas quedan con la misma misión — eso es válido (caso
  multi-tab trabajando juntas en un spec). NO bloquear.
- ¿El timeout de 30 s del toast es configurable? v1: hardcoded;
  configurable en Settings → v2.
- ¿La ventana de "candidate pending" para el guard de Engage AOM (10
  min) es la correcta? Si el usuario crea un spec, hace otras cosas, y
  vuelve a engageagar 30 min después, ¿debería seguir proponiéndolo?
  Recomendación: 10 min mantiene la "frescura" del wow moment;
  ajustable post-feedback.
- ¿Debe el toast tener un tercer botón `[Open spec]` para previsualizar
  antes de aceptar? v1: no (Goal snippet ya da contexto suficiente).
  Reconsiderar si el feedback dice que la gente acepta a ciegas.
