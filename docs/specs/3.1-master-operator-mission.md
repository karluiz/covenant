# 3.1 — Master Operator: mission attach + per-tab posture

## Goal

Let the user attach a mission spec (markdown file) to a tab and see/ flip the operator's 
posture for that tab in one place — so the master operator's "boundaries" are visible and 
editable from the tab itself, not buried in global settings.

> Status note: the engine is largely built. Persona + hard
> constraints (`crates/app/src/operator.rs`), safety blocklist
> (`crates/app/src/safety.rs`), Live toggle (M-OP3), decision
> thresholds (M-OP4), and mission Rust scaffolding
> (`set_mission` / `clear_mission` / `get_mission` at
> `operator.rs:358-396`) all exist. **What's missing is the UI
> surface for mission + per-tab posture.** This spec closes that gap.

## Out of scope

- **Editing the persona/charter from the mission overlay** — that
  belongs in `⌘,` (Settings) and is already there. The mission is
  a per-session scope, not a global identity edit.
- **Multi-operator support** — see `3.2-multi-operator.md`. This
  spec assumes a single master operator.
- **AOM minimalist mode** — see `3.4-aom-minimalist-ui.md`. Mission
  attach can happen inside AFK or normal UI; AFK rendering is
  separate.
- **Mission file authoring helpers** — no template insertion, no
  validation of spec structure. The user writes markdown; the
  operator consumes it as opaque context.
- **New backend commands beyond what `operator.rs` already exposes**
  — `set_mission`, `clear_mission`, `get_mission` are sufficient.
  If something is missing, ESCALATE; do not add new commands
  silently.
- **Mission persistence across app restarts** — see open questions.
- **Replacing the global Live toggle in Settings** — global toggle
  stays; per-tab posture is an *override*, not a replacement.

## Acceptance criteria

- [ ] `⌘M` opens a "Mission" overlay scoped to the active tab.
- [ ] Overlay shows: current mission (filename + path) or "No
      mission attached"; "Attach…" button opens a file picker;
      "Detach" button clears it.
- [ ] Attaching calls `set_mission(session_id, path)`; failures
      (missing file, permission denied) surface as toasts via the
      existing toast system.
- [ ] When a mission is attached, the tab shows a small chip
      (filename truncated) next to the tab title.
- [ ] Per-tab posture row in the overlay: three states —
      `Suggest` / `Live` / `AOM-excluded` — reflecting the existing
      per-tab live toggle and `aom_excluded` flag. Switching states
      calls existing commands; no new IPC.
- [ ] `Esc` closes the overlay; outside-click closes it.
- [ ] `npx tsc --noEmit` passes; `cargo check -p covenant` clean.

## File boundaries

- **Create**:
  - `ui/src/operator/mission.ts` — `MissionOverlay` class,
    open/close/toggle (≤ 220 lines).
- **Touch**:
  - `ui/src/main.ts` — `⌘M` shortcut + Esc handling +
    instantiation. Max 25 lines added.
  - `ui/src/operator/panel.ts` — link to mission overlay from
    operator panel (the existing right-click menu /
    ⌘O surface). Max 30 lines added.
  - `ui/src/api.ts` — wrap `set_mission`, `clear_mission`,
    `get_mission`. Max 40 lines.
  - `ui/src/styles.css` — overlay styles. Max 140 lines, appended
    at file end. Reuse `--bg-overlay`, `--border`, `--accent`,
    `--muted`, `--bg-panel`.
  - `ui/src/tabs/manager.ts` — render mission chip in tab strip.
    Max 25 lines added; do NOT alter drag/reorder/group logic.
- **DO NOT touch**:
  - `crates/app/src/operator.rs`, `aom.rs`, `safety.rs`,
    `settings.rs` — backend is complete for this spec.
  - `crates/app/src/lib.rs` — no new Tauri commands needed.
  - `ui/src/aom/`, `ui/src/recall/`, `ui/src/blocks/`,
    `ui/src/settings/` — orthogonal surfaces.

## Complexity

`small` — 1 AOM session, ~3-5 hours. UI-only feature on top of
existing backend.

- **Chip placement**: tab strip vs. status bar (3.7). If 3.7 ships
  first, prefer status bar; else tab strip. Agent: ESCALATE if
  ordering is unclear at run time.
