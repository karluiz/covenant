# 3.8 ‚Äî Convergence Mode

## Goal

A "command room" overlay that renders every terminal session as a
live tile so the user can see ‚Äî at a glance, without tab-hopping ‚Äî
what each agent executor is doing, what state it's in, and what
the operator decided last for it.

## Out of scope

- **Embedding inside AFK mode (3.4)** ‚Äî the long-term plan is for
  Convergence Mode to become AFK's primary view, with the
  chronological decisions feed demoted to a sidebar. v1 ships the
  overlay standalone so we can validate the metaphor before
  coupling it to AFK. The embedding is its own spec.
- **Multi-operator support** ‚Äî see `3.2-multi-operator.md`. v1
  assumes a single master operator across all tiles.
- **Agent-to-agent messaging / cross-tile orchestration UI** ‚Äî
  belongs in 3.2's surface, not here.
- **Editing agent prompts from a tile** ‚Äî tiles are read-only
  observability; user clicks through to the tab to act.
- **Tile drag / reorder / custom layout** ‚Äî fixed CSS-grid
  auto-flow. Tile order = tab order.
- **Recording / replay / scrubbing** ‚Äî separate spec.
- **Vendor detection (claude / copilot / opencode badge)** ‚Äî
  useful, but it's a fragile heuristic worth its own spec. v1
  treats every tab equally.
- **Filtering / hiding tiles** ‚Äî v1 shows all tabs. Per-tab "hide
  from Convergence" is v2.
- **Tile virtualization** ‚Äî v1 caps usable density at ~12 tiles
  before scroll kicks in; virtualization comes when someone
  actually has 30 tabs.
- **New per-tile shortcuts** ‚Äî only the overlay-level shortcut
  (`‚åò‚áßO`) and `Esc`. No tile-focused keynav in v1.
- **Convergence Mode as a marketing surface** ‚Äî tiles show real
  state always; they do not hype features.

## Acceptance criteria

- [ ] `‚åò‚áßO` toggles the Convergence Mode overlay (full-window,
      z-index above tab strip and sidebar ‚Äî same layering posture
      as AFK).
- [ ] Layout: CSS grid `auto-fit, minmax(280px, 1fr)`, 16 px gap,
      24 px outer padding. No JS layout calculation.
- [ ] Each tile shows, in this order top-to-bottom:
      (1) tab title (truncated) + the tab's existing color stripe;
      (2) status pill ‚Äî one of `idle` / `working` /
      `awaiting-input` / `blocked` / `operator-thinking`;
      (3) last operator decision (action label + one-line
      rationale, truncated to two lines);
      (4) last command (truncated) + last non-empty output line
      preview (ANSI-stripped, truncated);
      (5) cost-this-session footer ONLY when the tab is enrolled
      in AOM ‚Äî `$0.42 / $5.00 budget`. Hidden otherwise.
- [ ] Tiles update within 1.5 s of underlying state changes (one
      backend snapshot poll per second is acceptable for v1).
- [ ] Clicking a tile focuses that tab and exits the overlay.
- [ ] `Esc` and the explicit "Exit" button both close the
      overlay.
- [ ] Empty state: when there are zero tabs, the overlay shows a
      single centered message ("No sessions") and the "Exit"
      button. The shortcut still toggles.
- [ ] Overlay respects active vibrancy mode (uses
      `--bg-overlay`, no hardcoded colors).
- [ ] Status pill color comes from existing tokens
      (`--accent`, `--muted`, plus `--danger` if it already
      exists; ESCALATE before adding a new color token).
- [ ] `npx tsc --noEmit` passes; `cargo check -p covenant` is
      clean; new Rust tests cover the snapshot builder's status
      classification.

## Complexity

`medium` ‚Äî 2-3 AOM sessions. Largest risk: the `convergence.rs`
aggregator. If the data needed to populate a tile is not already
exposed by `operator.rs` / `aom.rs` / `session`, that's a signal
Convergence is reaching past its surface ‚Äî ESCALATE rather than
adding new fields to those modules.

## Open questions

- **Update mechanism**: snapshot poll (1 s tick from frontend)
  vs subscribing to existing per-session events and rebuilding
  client-side. Recommend snapshot poll for v1 (one endpoint,
  zero coupling to the event bus). ESCALATE if user prefers the
  push model ‚Äî that changes the file boundaries above.
- **Tile sort order**: tab position (default, predictable) vs
  most-recent-activity-first (livelier but jarring). v1 = tab
  position; mark "activity sort" as v2.
- **AFK embedding (v2)**: when Convergence Mode moves inside
  AFK, does it *replace* the current AFK feed or sit beside it
  as a toggleable view? Answer determines whether AFK gains a
  view-switcher or absorbs Convergence wholesale. Out of scope
  here, but flag the decision before starting v2.
- **Vendor detection follow-up**: detecting which executor
  (claude / copilot / opencode) is running gives tiles much
  richer status, but the heuristic is fragile and worth its own
  spec. ESCALATE if the user wants this in v1 instead of
  follow-up.
- **Cost footer when AOM is global-on but the tab is excluded**:
  show `excluded from AOM` chip or hide cost row entirely?
  Default: hide cost row, show no chip.

## Resolved decisions (brainstorm 2026-05-03)

- **Update mechanism**: snapshot poll, 1 s tick from frontend. Single
  `get_convergence_snapshot` Tauri command. No event-bus coupling.
- **Tile sort order**: tab position. "Activity sort" deferred to v2.
- **Cost footer when tab excluded from AOM**: hide row entirely, no
  chip. Spec default stands.
- **AFK embedding, vendor detection**: out of scope, per spec.
