# 3.2 — Multi-Operator (gating spec)

> **This is a gating spec, not an actionable one.** The vision
> explicitly defers multi-operator to "next phase" after the master
> operator (3.1) has been validated. AOM should NOT execute this
> spec until the open questions below are resolved by the user and
> a follow-up actionable spec is written.

## Goal

Allow multiple operators with distinct capabilities, autonomy
levels, and personas to coexist — with a surface that makes
orchestration legible to the user.

## Out of scope

This entire spec is intentionally non-actionable. Concretely out
of scope until the gating questions are resolved:

- **Implementation** — agent must NOT write code from this spec.
- **Schema migrations** — `aom_sessions`, `operator_state`, etc.
  remain single-operator.
- **UI redesign of the operator panel** — current single-operator
  panel stays.
- **Inter-operator messaging / handoff protocol** — premature
  until the orchestration model is decided.
- **Per-operator API key / model routing** — depends on
  orchestration outcome.

## Acceptance criteria

(Intentionally empty — this spec gates further work.)

## Complexity

`large` — multiple nights once promoted, but currently `gating`.
Break into at least 3 actionable specs before passing any of it to
AOM:

1. Orchestration runtime + arbitration policy.
2. Per-operator state (persona, quota, capabilities) + persistence.
3. UI surface (operator list, active-operator indicator,
   handoff visualization).

## Open questions

These MUST be resolved by the user before this spec is promoted:

- **Orchestration model**: round-robin, leader-elect (one
  "primary" with delegates), capability-routing (operator selected
  per decision-type), or user-pinned (one operator per tab)?
- **Quota model**: per-operator daily token budget, or a shared
  pool with weighted priority?
- **Conflict resolution**: when two operators want different
  actions for the same prompt, who wins? Vote? Persona-with-priority?
  Defer to user?
- **Visualization**: how does the user see "operator X is currently
  thinking about tab 2 while operator Y is asleep on tab 5"?
  AFK mode (3.4) implications?
- **Capability declarations**: do operators declare what they CAN
  do (e.g., "git only", "test only"), or are they fully general
  with persona-driven differences?
- **Backwards compatibility**: how do existing single-operator
  AOM session rows migrate?
