# 3.22 — Operator parallel task slots

## Goal

Let one Operator persona supervise multiple active teammate tasks at once
without corrupting context, fighting over one terminal, or hiding what is
happening from the user.

The mental model is:

```text
Operator persona
  ├─ Task slot A → isolated tab/session/worktree A
  ├─ Task slot B → isolated tab/session/worktree B
  └─ Task slot C → isolated tab/session/worktree C
```

The Operator remains the human-facing persona, budget owner, and policy
surface. Each active task gets its own **slot** with isolated runtime
state, session binding, and conflict checks. This turns the teammate
panel from "operator is on one task" into "operator is supervising N
bounded workers".

## Current behavior / problem

Today the teammate runtime is single-task per operator:

- `crates/app/src/teammate/runtime.rs` stores
  `HashMap<OperatorId, OperatorState>`.
- `OperatorState::OnTask { task, session }` can hold exactly one task.
- `TeammateRuntime::start_task` rejects a second task with
  `operator already on task`.
- `ui/src/teammate/panel.ts` defaults confirmed `do` tasks to the active
  tab unless `localStorage.covenant.teammate.confirm-target = "spawn"`.

This is safe, but it blocks the high-value workflow where a user asks
one trusted Operator to work through two or three independent tasks in
parallel.

## Product decision

Parallelism is **opt-in and slot-limited**.

- Default `max_parallel_tasks = 1` to preserve existing behavior.
- User/operator setting can raise it to `2` or `3`.
- Hard cap in the first implementation: `3` active task slots per
  operator.
- A `do` task that would exceed the cap is queued, not rejected.
- A confirmed task should prefer a **dedicated tab/session**. Driving
  multiple tasks into the same PTY is not allowed unless the user
  explicitly overrides after a warning.
- Worktree isolation is recommended for code-writing tasks and can be
  automatic once the branch/worktree plumbing exists.

## Out of scope

- **Multiple independent Operator personas collaborating**. This spec is
  one Operator with multiple task slots. Multi-operator arbitration is a
  separate 3.2 follow-up.
- **One executor TUI handling multiple concurrent objectives**. Claude,
  Codex, Copilot, Hermes, and Pi sessions should still receive one task
  objective at a time.
- **Automatic merge resolution**. Conflict detection may block or queue a
  task; resolving file conflicts is a later spec.
- **Unbounded background autonomy**. Slots are capped, budgeted, visible,
  and cancellable.
- **Remote worker fleet / cloud execution**. First pass uses local tabs,
  local sessions, and optional git worktrees.

## Acceptance criteria

### Runtime + persistence

- [ ] Each Operator has a configurable `max_parallel_tasks` persisted in
      the operator settings, default `1`, allowed range `1..=3`.
- [ ] `TeammateRuntime` supports multiple active task slots per operator
      instead of a single `OnTask` state.
- [ ] Confirming a task when active slots are available starts a new slot;
      confirming a task when slots are full marks the task as queued.
- [ ] Runtime state can answer: active count, queued count, slot id,
      task id, session id, started_at, and last heartbeat per operator.
- [ ] App restart reconstructs active/queued task state from
      `teammate_tasks` rows and their `spawned_session` values as far as
      possible. Missing sessions become `blocked` or `queued` with a
      user-visible note.

### Scheduling behavior

- [ ] Task statuses include a queued state (or an equivalent persisted
      representation) so full slots do not look like failures.
- [ ] Scheduler starts queued tasks FIFO when a slot completes, is
      cancelled, or becomes blocked and releases its execution slot.
- [ ] The scheduler never dispatches two active tasks into the same
      `SessionId` by default.
- [ ] A confirmed `do` task defaults to spawning/owning a dedicated tab
      when parallel slots are enabled. The old active-tab injection path
      remains available for `max_parallel_tasks = 1` and explicit manual
      confirm flows.
- [ ] `watch` and `review` tasks may share observation context, but any
      task that writes code still requires an owned execution session.

### Isolation and conflict safety

- [ ] Each slot records a `TaskScope` path set. Before dispatch, the
      scheduler compares it with active tasks for the same repo/operator.
- [ ] If two active tasks declare overlapping `scope.paths`, the newer
      task is queued or blocked with a conflict reason.
- [ ] If a task has empty/unknown path scope and targets the same repo as
      another active code-writing task, the UI marks it as "unscoped" and
      requires either explicit user confirmation or a dedicated worktree.
- [ ] Optional worktree mode can create a new git worktree per task using
      a deterministic branch/worktree name. If worktree creation fails,
      the task falls back to queued/blocked, not shared-session dispatch.
- [ ] Existing safety paths for destructive command checks, operator live
      toggles, and secret masking are unchanged.

### UI / user visibility

- [ ] Teammate header shows `N / max` active slots for the selected
      operator, not only one "working" ring.
- [ ] Tasks tab groups tasks into: Active, Queued, Blocked, Done, and
      Cancelled.
- [ ] Each active task row shows its slot number, attached tab/session,
      executor, repo/worktree label, status, elapsed time, and last
      heartbeat.
- [ ] Clicking a task focuses its owned tab/session.
- [ ] User can pause/cancel a queued task before it starts.
- [ ] User can stop an active slot, which marks the task blocked or
      cancelled and frees the slot.
- [ ] If slots are full, proposal cards say "Queued — waiting for slot"
      instead of surfacing `operator already on task`.

### Budgeting / controls

- [ ] Cost accounting remains per task (`cost_usd_cents`) and also rolls
      up per operator across all active slots.
- [ ] An operator-level budget cap applies across all slots; reaching the
      cap pauses queued starts and blocks active autonomous decisions that
      would incur additional LLM cost.
- [ ] The settings UI explains that parallel slots multiply token and
      command-execution risk.

### Tests

- [ ] Unit tests for `TeammateRuntime`: start up to cap, reject/queue over
      cap, finish frees a slot, reset clears all slots, attach session is
      unique per active slot.
- [ ] Storage tests for queued/active task rows and restart
      reconstruction.
- [ ] UI tests for teammate header slot count, queued proposal copy, and
      click-to-focus task rows.
- [ ] Regression test: with `max_parallel_tasks = 1`, existing single-task
      behavior and friendly error handling still work.

## Proposed data model

Prefer additive migrations.

### `operators`

Add:

- `max_parallel_tasks INTEGER NOT NULL DEFAULT 1`

If operator settings currently serialize only through registry JSON, add
the field there instead of SQLite; keep the Tauri/API shape stable for
older operator rows by defaulting missing values to `1`.

### `teammate_tasks`

Add if needed:

- `slot_id TEXT NULL` — stable id for the active slot assignment.
- `queued_at_unix_ms INTEGER NULL`
- `started_at_unix_ms INTEGER NULL`
- `blocked_reason TEXT NULL`
- `repo_root TEXT NULL`
- `worktree_path TEXT NULL`

If adding a `queued` status is too invasive, use `status='draft'` plus
`queued_at_unix_ms` only as a temporary migration step, but the final UI
should expose `queued` as a first-class state.

### Runtime shape

Replace the single `OperatorState::OnTask` with something like:

```rust
pub enum OperatorState {
    Idle,
    Pinned { session: SessionId },
    Supervising { slots: Vec<TaskSlot>, queued: Vec<TaskId> },
}

pub struct TaskSlot {
    pub id: SlotId,
    pub task: TaskId,
    pub session: Option<SessionId>,
    pub repo_root: Option<PathBuf>,
    pub worktree_path: Option<PathBuf>,
    pub started_at_unix_ms: u64,
    pub last_heartbeat_unix_ms: Option<u64>,
}
```

`Pinned` remains for manual operator pinning. Once the first task starts,
`Pinned` can either become `Supervising` or coexist as a non-task binding,
but the UI must not imply that the pinned tab is the only active work.

## Scheduling rules

1. Confirmed task enters `queued` or `active` based on slot capacity.
2. For `do` tasks, resolve an execution target:
   - existing dedicated task session if already attached;
   - new spawned tab/session;
   - optional new git worktree + spawned tab;
   - explicit user-selected active tab only if no other active task owns
     that session.
3. Run conflict checks before sending bytes to the executor.
4. Attach session to task, bind operator to that tab, enable single-tab
   live mode, then inject the task prompt.
5. Slot remains active until task completes, is cancelled, blocks, or the
   user stops the slot.
6. When a slot frees, scheduler starts the oldest queued task that passes
   conflict and budget checks.

### Do not touch without a follow-up spec

- `crates/app/src/operator.rs` core autonomous prompt-answering policy,
  except for reading the task/session binding it already consumes.
- Provider dispatch paths outside teammate/operator task scheduling.
- Secret masking and safety filters except to reuse them.
- Cross-operator arbitration / 3.2 multi-operator design.

## Rollout plan

### Phase 1 — Queue without parallel execution

- Add `queued` state and scheduler bookkeeping.
- Keep `max_parallel_tasks = 1` everywhere.
- Replace `operator already on task` with queued UX.

### Phase 2 — Parallel slots with dedicated sessions

- Add `max_parallel_tasks` setting.
- Start up to 2–3 slots concurrently.
- Always spawn/own a dedicated tab/session for parallel `do` tasks.
- Add UI slot count and task grouping.

### Phase 3 — Conflict detection + worktree isolation

- Compare scopes across active tasks.
- Add warning/blocking behavior for overlapping paths.
- Add optional worktree-per-task mode.

### Phase 4 — Budget and recovery polish

- Per-operator aggregate cost caps.
- Restart reconstruction.
- Slot heartbeat/staleness detection.
- Stop/pause/resume controls.

## Complexity

`large` — should not be handed to AOM as one monolithic task. Split into
at least the four phases above, with Phase 1 small/medium and Phases 2–4
medium/large depending on worktree automation.

## Open questions

- Should `max_parallel_tasks` live on the Operator profile, global
  teammate settings, or both (global default + per-operator override)?
- Should worktree-per-task be automatic for all code-writing tasks, or a
  per-operator setting?
- What is the default dispatch target once parallelism is enabled:
  always new tab, or active tab for the first slot and new tabs after
  that?
- When a task blocks, does it keep its slot until the user resolves it,
  or release the slot and move to a blocked queue?
- Should `review` tasks count against the same slot cap as `do` tasks?
- What is the user-facing name: "parallel tasks", "slots", "workers",
  or "lanes"?
