# 3.14 — Escalation Visibility

## Goal

When operators escalate across multiple tabs in AOM mode, the user can see at a glance (a) which mission/project each escalated tile is for, and (b) which tabs are escalated without opening convergence first. No more digging into a session to know which project needs attention.

## Out of scope

- OS-level notifications on escalation (3.6 / future spec).
- Status bar summary "N tabs escalated" (future spec).
- Per-operator rate limiting / budget — separate concern.
- Filter / hide non-escalated tiles in convergence — v1 still shows all.
- Sound / haptic alerts.
- Persistent escalation log across app restarts.

## Acceptance criteria

### Convergence tile — mission line

- [ ] Each tile renders a mission line below the header (above status pill) when the session has a mission attached. Format: `📍 <mission-name>` where `<mission-name>` is the basename of the mission path stripped of `.md` extension (e.g. `3.12-operators-experience-and-level.md` → `3.12-operators-experience-and-level`). Truncate at 40 chars with ellipsis.
- [ ] When no mission is attached, the line is hidden (no placeholder, no layout shift).
- [ ] Mission text uses `var(--muted)` color, font-size 11px, no new tokens.
- [ ] Backend snapshot supplies `mission_name: Option<String>` on `ConvergenceTileState` — derived from the most recent decision row's `mission_path`. NULL if no decision or no mission. No new public surface on `OperatorWatcher`.

### Tab chip — escalation indicator

- [ ] When a session's most recent operator decision was `escalate` AND no new bytes arrived since (i.e. operator is genuinely blocked, matching the convergence `blocked` status logic), the corresponding tab chip renders a small red dot indicator. Position: top-right of the chip, 8px diameter, `var(--danger)` (or `var(--accent)` if `--danger` not defined).
- [ ] The dot pulses subtly (2-second loop, 0.7→1.0 opacity) so it draws the eye without being disruptive.
- [ ] The dot disappears when the session unblocks (status flips out of blocked, e.g. via convergence reply or manual user input in the tab).
- [ ] Dot updates within 1.5s of state change (same cadence as convergence — reuse existing snapshot poll or tab-state subscription).

### No regressions

- [ ] Tabs without escalation: visually identical to today.
- [ ] Convergence tiles for sessions without mission: visually identical to today (no extra blank line).
- [ ] `cargo check`, `cargo test`, `npx tsc --noEmit` all clean.

## File boundaries

### Create
None.

### Modify
- `crates/app/src/convergence.rs` — add `mission_name: Option<String>` to `ConvergenceTileState`; populate from `OperatorDecisionRow.mission_path` (already pulled).
- `ui/src/api.ts` — add `mission_name` field to `ConvergenceTileState` typescript type.
- `ui/src/convergence/tile.ts` — render the mission line.
- `ui/src/styles.css` — append minimal styles for `.convergence-tile__mission` and `.tab-chip__escalation-dot` (≤60 lines).
- `ui/src/tabs/manager.ts` — add the escalation dot DOM element, wire its visibility to a per-session blocked state.
- `ui/src/main.ts` — minimal: if a new shared blocked-state subscription is needed for tab chips, instantiate it.

### DO NOT touch
- `crates/app/src/operator.rs`, `aom.rs`, `safety.rs`, `storage.rs` — read-only consumers. Mission name comes from data already pulled.
- `crates/agent/`, `crates/blocks/`, `crates/session/`, `crates/pty/`.

## Complexity

`small` — display-only, all data already flowing. Single AOM session suffices.

## Resolved decisions

- **Tab chip blocked-state source**: new lightweight Tauri command `get_blocked_session_ids() -> Vec<SessionId>`, polled at 1 Hz from `TabManager` (independent of convergence overlay visibility). Convergence snapshot only runs while overlay is open, so it cannot drive tab-chip indicators.
- **Mission name shortening**: basename of mission path minus `.md` extension. Truncate to 40 chars in the UI.
