# 3.12 — Operators Experience and Level

## Goal

Gamify the operator workflow: each completed task grants XP, and operators level up linearly so the user can watch progress accumulate AFK-style across long autonomous runs.

## Out of scope

- Prestige systems, skill trees, or branching/non-linear level paths
- Visual animations, badges, achievement popups, or celebratory level-up effects
- Configurable XP multipliers, bonus events, or time-limited XP boosts
- Per-operator leaderboards or cross-operator comparisons
- Rewards, unlocks, or behavior changes tied to level (level is cosmetic only)
- Retroactive XP backfill for tasks completed before this feature ships

## Acceptance criteria

- [ ] Operator XP total and current level render in the terminal UI at all times and update immediately after a task completes (no reload, no manual refresh).
- [ ] When a task is marked complete, the backend awards the XP defined for that task type, persists the new total to the operator row in SQLite, recomputes level via the linear threshold, and returns the updated values in the success response.
- [ ] Crossing one or more level thresholds in a single XP award increments level by the exact number of thresholds crossed; subsequent API responses and UI reads reflect the new level.
- [ ] XP and level survive app restart (persisted, not in-memory).
- [ ] An operator with zero XP renders as level 1 with 0 XP without special-casing in the UI.

## File boundaries

- **Create**: `crates/agent/src/xp.rs` (XP table + level calculation, ≤ 80 lines)
- **Touch**: operator persistence layer (SQLite migration adding `xp INTEGER NOT NULL DEFAULT 0`), task-completion handler (award XP on success), operator panel UI component (render XP + level badge)
- **DO NOT touch**: safety/blocklist code, AOM session loop, cost-cap logic — XP is purely additive metadata

## Complexity

`small` — single migration, one pure function for level math, one UI binding. Fits in one AOM session.

## Decisions (locked)

- **"Task" = Operator decision.** Three action kinds; XP table:
  - `reply` → **10 XP**
  - `escalate` → **25 XP**
  - `wait` → **1 XP**
- **Linear threshold:** **100 XP per level**. `level = floor(xp / 100) + 1`. Level is purely cosmetic (no behavior change).
- **Badge placement:** rendered next to the operator avatar — both the per-tab `tab-op-chip` (always visible while the tab is open) and the per-row chip in the ⌘O Operator panel. Format: `Lv {N}` with `{xp} XP` in the title attribute.
- **Live updates:** backend emits `operator-xp-updated` `{ operator_id, xp }` after each decision is persisted; tab manager + operator panel update their caches in-place without a full reload.

## Open questions

None — all design questions resolved.
