# 3.24 — Tab Styles

## Goal

Let users compose tab appearance from atomic properties (shape, background, active indicator) via Settings → Experimental, instead of being locked into four monolithic presets. Inspired by Discord's customization approach — turn every visual knob independently.

## Out of scope

- Per-tab or per-group gradient/gradient assignment (each tab gets its own colors). This spec does global tab bar styling only.
- Per-tab gradient background. A global tab background gradient is in scope; assigning different gradients per individual tab is not.
- Tab animations or transitions beyond what currently exists (`transition: background 0.12s`).
- Drag-and-drop tab rearrangement (already works).
- A visual "preview" canvas of the tabbar inside settings. The user saves and sees the real tabbar update live.
- The existing Forge / Glass / CRT presets remain untouched and separate. This spec does not refactor them.

## Acceptance criteria

- [ ] A new `tab_styles` toggle in Settings → Experimental enables the custom tab styling UI.
- [ ] When `tab_styles` is on, a "Custom tab style" section appears in Settings → Appearance with shape, background mode, indicator, height, and gap controls.
- [ ] Tuning any control updates the tabbar instantly (no restart). Changes persist to `config.json`.
- [ ] The classic default ("no body class") is the base; custom overrides sit on top as CSS custom properties on `:root`.
- [ ] Disabling `tab_styles` removes all custom overrides and the control group disappears; the tabbar snaps back to the chosen preset (Classic / Forge / Glass / CRT).
- [ ] `cargo test --workspace` and `npm test` pass with no regressions.

## Design

### Experimental config (Rust)

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TabStylesConfig {
    pub enabled: bool,
    pub shape: TabShape,
    pub bg_mode: TabBgMode,
    pub bg_gradient: Option<[String; 2]>,  // start + end hex, None = flat
    pub indicator: TabIndicator,
    pub height: TabHeight,
    pub gap: TabGap,
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(rename_all = "lowercase")]
pub enum TabShape {
    #[default]
    Rectangle,
    Rounded,
    Lofted,
    Pill,
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(rename_all = "lowercase")]
pub enum TabBgMode {
    #[default]
    Solid,        // opaque
    Translucent,  // semi-transparent
    Off,          // fully transparent, just text + indicator
    Gradient,     // uses bg_gradient
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(rename_all = "lowercase")]
pub enum TabIndicator {
    #[default]
    Stripe,    // current top-edge accent stripe
    Underline, // bottom-edge underline
    LeftBar,   // vertical left edge
    Dot,       // small circle
    Glow,      // box-shadow glow
    Border,    // colored outline
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(rename_all = "lowercase")]
pub enum TabHeight {
    #[default]
    Normal,  // 30px
    Compact, // 26px
    Spacious,// 36px
}

#[derive(Debug, Clone, Serialize, Deserialize, Default)]
#[serde(rename_all = "lowercase")]
pub enum TabGap {
    #[default]
    Normal, // 4px
    Tight,  // 2px
    Loose,  // 8px
}
```

Add to `ExperimentalConfig`:
```rust
#[serde(default)]
pub tab_styles: TabStylesConfig,
```

### Frontend

**New body class:** `body.tab-style-custom` — activated when `experimental.tab_styles.enabled` is true. Its presence tells CSS "use custom variables." A single small CSS file (`tab-themes/custom.css`) scopes selectors under this class.

**CSS variables mapped from config:**

| Config field | CSS variable | Example values |
|---|---|---|
| `shape` | `--tab-custom-radius` | `0px`, `6px`, `12px`, `20px` |
| `bg_mode` | `--tab-custom-bg` | flat color, `transparent`, or gradient string |
| `indicator` | `--tab-custom-indicator` | `stripe`, `underline`, `left-bar`, `dot`, `glow`, `border` |
| `height` | `--tab-custom-h` | `26px`, `30px`, `36px` |
| `gap` | `--tab-custom-gap` | `2px`, `4px`, `8px` |

Each variable has a sensible default so `body.tab-style-custom` works even without explicit overrides (defaults match the Classic preset).

**Settings UI:** When Experimental → Tab Styles is toggled on, a "Custom tab style" card appears in the Appearance section with:
- Shape: radio group (Rectangle / Rounded / Lofted / Pill)
- Background: radio group (Solid / Translucent / Off / Gradient) + two color pickers when Gradient is selected
- Active indicator: radio group (Stripe / Underline / Left bar / Dot / Glow / Border)
- Height: radio group (Compact / Normal / Spacious)
- Gap: radio group (Tight / Normal / Loose)

**Live application** (`applyCustomTabStyle`): Called on settings save and on init. Reads `config.experimental.tab_styles`, toggles `body.tab-style-custom`, and calls `document.documentElement.style.setProperty(...)` for each CSS variable. If disabled, removes the body class and clears the custom properties — the tabbar falls back to whatever preset radio button is selected.

### CSS approach (`custom.css`)

```css
body.tab-style-custom .tab-btn {
  border-radius: var(--tab-custom-radius, 0px);
  background: var(--tab-custom-bg, var(--tab-bg-rest));
  height: var(--tab-custom-h, 30px);
}
```

For each indicator variant, a `body.tab-style-custom` + data-attribute or class combo targets `::before` / `::after` to paint the indicator shape. For example:

```css
body.tab-style-custom[data-tab-indicator="underline"] .tab-btn.active::before {
  top: auto;
  bottom: 0;
  height: 2px;
  border-radius: 0;
}
body.tab-style-custom[data-tab-indicator="dot"] .tab-btn.active::before {
  top: 50%;
  left: 4px;
  width: 6px;
  height: 6px;
  border-radius: 50%;
  transform: translateY(-50%);
}
```

(Alternatively, use a data attribute on `body` to avoid class toggling for each variant.)

## Open questions

- Should gradient mode support 3+ color stops, or is 2 enough for v1?
- Should `bg_mode` = `Off` also hide the tab border, or keep a subtle border for hit-area affordance?
- Should the existing tab style radio buttons be hidden when `tab_styles.enabled` is true, or stay visible but grayed out?
- Does the gradient apply per-tab (flat fill per tab, but the bar as a whole is gradient-themed) or as a continuous gradient across the entire tab bar? Per-tab gradient (each tab gets the full gradient as its background) is simpler and matches the current `--tab-bg-rest` model — but a continuous gradient across the tabbar could look more polished. This is a visual decision.
