# 3.13 — Operator Learning (Decision Memory)

## Goal

When the operator escalates a decision and the user replies to it in convergence mode, that reply (a) unblocks the escalated session immediately and (b) is captured as a reusable memory so the operator stops escalating the same trivial pattern twice. The user wants to write each rule once, in the place they're already replying — convergence — not by editing specs by hand on every escalation.

## Depends on

- **3.8 Convergence Mode** — the reply channel. Without convergence there is no coherent place for the user to "answer" an escalated operator. This spec assumes 3.8 is shipped (or shipping in the same window).

## Out of scope

- Generic long-term memory for the operator's world-model (specs, mission state, code knowledge)
- Cross-user / cross-machine sharing of learned decisions
- Automatic un-learning, decay, or reinforcement scoring — memories are explicit, edited by hand if wrong
- Fine-tuning, embeddings retraining, or any model-side learning
- Learning from non-escalation events (success patterns, command history, etc.)
- A full UI for browsing/editing memories (a JSONL file the user can open is enough for v0)

## Acceptance criteria

- [ ] In convergence mode, an `ESCALATED` operator shows its open question and accepts a free-text reply from the user.
- [ ] Submitting the reply (a) injects the answer back into the escalated session so the operator resumes immediately without the user touching the spec, and (b) appends a memory entry `{pattern, decision, rationale, scope, created_at}` to `~/.karl/operator-memory.jsonl`.
- [ ] The convergence reply UI lets the user pick the scope at submit time (`global`, `mission:<id>`, or one-shot — one-shot does NOT persist as memory).
- [ ] On every operator decision step, relevant memories are retrieved (tag/keyword match over `pattern` and `scope`) and injected into the operator prompt under a stable `## Learned decisions` section so prompt caching is preserved.
- [ ] If a learned memory matches the current open question, the operator applies it instead of escalating, and logs `applied_memory: <id>` in the decision row so the user can audit.
- [ ] The user can open `~/.karl/operator-memory.jsonl` and edit/delete entries by hand; the operator picks up changes on next decision (no restart).
- [ ] An operator with zero memories behaves exactly as today (no regression on escalation behavior).

## File boundaries

- **Create**: `crates/agent/src/memory.rs` (store + retrieval, ≤ 150 lines), `~/.karl/operator-memory.jsonl` (runtime, not committed)
- **Touch**: operator decision loop (inject memories into prompt, log `applied_memory`), escalation resolution handler (route convergence reply back into the session), convergence-mode UI (reply box + scope selector on `ESCALATED` tiles)
- **DO NOT touch**: safety blocklist, cost-cap logic, AOM session loop core — learning is advisory, never overrides hard guardrails

## Complexity

`medium` — 2 sessions. One for the store + retrieval + prompt injection, one for the UI capture flow and audit log wiring.

## Open questions

- **Scope granularity**: global, per-mission, or per-spec-type? Recommendation: start with two scopes — `global` and `mission:<id>` — and let the user pick at capture time. Confirm before building.
- **Retrieval method**: keyword/tag match (cheap, deterministic) vs. embedding similarity (better recall, infra cost). Recommendation: tags + substring match for v0; embeddings deferred to a later spec if recall is poor.
- **Conflict resolution**: if two memories match and disagree, does the operator pick the most recent, the most specific scope, or escalate? Default proposal: most-specific-scope wins, ties broken by recency, never silently — log which memory won.
- **Capture trigger**: the convergence reply itself is the trigger; the scope selector (`global` / `mission` / `one-shot`) is how the user opts out of persistence. Confirm this is enough or whether a separate "save as rule" button is also wanted.
- **Reply → session injection**: how exactly does the reply reach the escalated session — as a synthetic user message in the operator's transcript, as a structured `resolution` event the operator handles, or as a direct write to the spec's open question? Recommendation: structured `resolution` event so the same path serves both the unblock and the memory write.
