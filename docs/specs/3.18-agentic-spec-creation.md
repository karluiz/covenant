# 3.18 — Agentic spec creation (chat-first authoring)

## Goal

Crear specs conversando con un agente en lenguaje natural, no llenando
seis textareas en blanco. El formulario actual (Goal / Out of scope /
Acceptance / File boundaries / Complexity / Open Qs) sigue existiendo
pero como **visualizador y editor de revisión**, no como punto de
entrada. El humano describe la idea en una o dos frases; el agente
hace 3–5 preguntas dirigidas y emite el markdown ya en nuestro
formato (`docs/specs/_template.md`).

El insight es que el spec es un **artefacto de salida del diálogo**,
no de entrada. La disciplina (forzar pensar en out-of-scope y file
boundaries antes de publicar) vive en el prompt del agente y en una
validación pre-publish, no en una UI vacía.

## Out of scope

- **Cambiar el formato de spec** — sigue siendo `_template.md`; el
  agente debe respetar las secciones existentes 1:1.
- **Rediseñar el editor de revisión** — los campos actuales se
  reusan tal cual; sólo cambia que ahora llegan pre-rellenados.
- **Spec drafting multi-spec en paralelo** — un draft activo a la
  vez. Si hay otro draft sin publicar, el chat lo retoma.
- **Importar specs externos** (markdown pegado, links a Linear/Notion)
  — eso es spec aparte si lo queremos.
- **Auto-publicar sin revisión humana** — el agente *propone*; el
  coordinador siempre pasa por el editor antes de `Publish`.
- **Cambios al detector 3.16 (`spec:candidate`)** — ortogonal.

## Acceptance criteria

- [ ] `⌘N` (o botón "New spec" en sidebar) abre un panel de chat con
      un prompt inicial: "¿Qué problema quieres resolver?".
- [ ] El agente hace entre 3 y 5 preguntas dirigidas cubriendo: goal,
      out-of-scope, acceptance, file boundaries, complexity. No más.
- [ ] Al terminar el diálogo, el agente emite un `.md` válido que
      cumple `_template.md` (todas las secciones presentes, no
      placeholders sin reemplazar) y abre el editor existente
      pre-rellenado.
- [ ] El editor permite editar libremente; `Publish` valida que las
      6 secciones estén no-vacías antes de escribir a
      `docs/specs/<id>-<slug>.md`.
- [ ] El id se sugiere como `3.<next>` leyendo el directorio; el
      coordinador puede sobrescribir.
- [ ] Si el coordinador cierra el chat a medias, el draft persiste y
      `⌘N` retoma desde la última pregunta contestada.
- [ ] Sigue existiendo entrada "Blank draft" para el caso 10% en
      que alguien quiera escribir a mano (mismo editor, sin chat).
- [ ] `cargo test -p covenant` pasa (no hay regresión).

## File boundaries

- **Create**:
  - `ui/src/spec-chat/` — UI del panel de chat (≤ 400 líneas total,
    máx 4 archivos: `panel.ts`, `dialogue.ts`, `emit.ts`, `state.ts`).
  - `crates/agent/src/spec_author.rs` — prompt + flujo del agente
    autor (≤ 250 líneas). Reusa `agent::dispatch()`, no llama a la
    API directo.
- **Touch**:
  - `ui/src/sidebar.ts` o equivalente — botón "New spec" (≤ 20 líneas).
  - `ui/src/commands.ts` (o cmd palette) — atajo `⌘N` (≤ 15 líneas).
  - Editor de specs existente — aceptar prop `initialDraft` (≤ 30
    líneas).
- **DO NOT touch**:
  - `crates/app/src/spec_detector.rs` (3.16) — detección sigue igual.
  - `_template.md` — el formato es contrato.
  - `crates/agent/src/safety.rs` — el spec author es read-only sobre
    el FS hasta que el humano publica.

## Complexity

`medium` (2–3 sesiones AOM). El backend del agente autor es chico;
la UI del chat panel y la integración con el editor existente es lo
que consume tiempo.

## Open questions

- **¿Streaming de las preguntas o batched?** Streaming se siente
  mejor pero cuesta más wiring sobre `agent::dispatch()`. Default
  propuesto: batched (una pregunta a la vez, no streaming token-level).
  → Escalar si el coordinador quiere streaming.
- **¿El agente puede mirar el repo para sugerir file boundaries?**
  Tentador (ej: detectar si la feature toca `crates/agent`), pero
  amplía el blast radius del prompt. Default propuesto: NO en v0,
  agregar en v2 si la fricción es alta.
- **¿Qué modelo?** Sonnet 4.6 es suficiente; no necesita Opus para
  emitir markdown estructurado. Confirmar con el coordinador.
- **¿Dónde vive el draft persistido?** Propuesta: `~/.covenant/spec-drafts/<ulid>.json`
  con `{messages, partialMd, lastUpdated}`. Confirmar antes de impl.
