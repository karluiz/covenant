# 3.7 — Status Bar

## Goal

Add a discrete bottom status bar that shows context derived from
the active tab's `cwd` — repo + branch (when in a git repo),
detected runtime (Node, Python, Go, Rust, …) — so the user knows
where they are without typing `pwd` or `git status`.

## Out of scope

- **Powerline-style multi-segment user-configurable bar** — fixed
  segments only (git, runtime). User config = on/off, not order.
- **Real-time stats** (CPU, RAM, network) — not this spec.
- **Notifications / alert badges in the bar** — see
  `3.6-os-notifications.md`.
- **Click actions on segments** — read-only v1. No "click branch
  to switch", no "click runtime to open settings".
- **Git status beyond branch name** — no ahead/behind counts, no
  dirty indicator. v2.
- **Worktree-aware display** — show whatever `git rev-parse` says
  about the cwd; don't try to detect worktree topology.
- **Showing the bar inside the AFK overlay** — AFK (3.4) has its
  own minimal header. Status bar is for normal mode only.
- **i18n** — English labels only. (Same posture as 3.5.)

## Acceptance criteria

- [ ] Bar is rendered at the bottom of the window, ~22 px tall,
      between the active terminal and the window's bottom edge.
- [ ] Segments render only when applicable:
      - `git`: `<repo-name> <branch>` with a small git icon.
      - `runtime`: detected runtime + version (e.g.,
        `node 20.11`, `python 3.12`, `rust 1.84`).
- [ ] Bar updates within 500 ms of a `CwdChanged` event for the
      active tab.
- [ ] Switching tabs re-renders the bar from the new active tab's
      cwd.
- [ ] Settings toggle (default ON) hides the bar entirely; when
      hidden, no detection runs.
- [ ] Bar respects vibrancy (`--bg-overlay` / `--bg-panel` token,
      no hardcoded colors).
- [ ] `npx tsc --noEmit` passes; `cargo check -p covenant` clean.
- [ ] Detection does not block the UI thread — runtime detection
      happens in the Rust backend on a worker.

## File boundaries

- **Create**:
  - `crates/app/src/context.rs` — `detect_git_context(cwd)` and
    `detect_runtime(cwd)` (≤ 200 lines). Pure functions; cache
    keyed by cwd with a small LRU.
  - `ui/src/status/bar.ts` — `StatusBar` class, subscribes to
    cwd events, renders segments (≤ 220 lines).
- **Touch**:
  - `crates/app/src/lib.rs` — register `get_dir_context(cwd)`
    Tauri command. Max 25 lines.
  - `crates/app/src/settings.rs` — add `status_bar_enabled: bool`
    (default `true`). Max 15 lines.
  - `ui/src/main.ts` — instantiate `StatusBar`, wire the
    cwd-change subscription. Max 30 lines.
  - `ui/src/api.ts` — wrap `get_dir_context`. Max 30 lines.
  - `ui/src/styles.css` — bar styles. Max 90 lines, appended at
    end. Reuse `--bg-panel`, `--border`, `--muted`, `--fg`.
  - `ui/src/settings/panel.ts` — single toggle. Max 20 lines.
- **DO NOT touch**:
  - `crates/agent/`, `crates/session/`, `crates/blocks/`,
    `crates/pty/`.
  - `ui/src/blocks/`, `ui/src/recall/`, `ui/src/operator/`,
    `ui/src/aom/`, `ui/src/structure/`, `ui/src/tabs/`.
  - The xterm.js mount or terminal rendering — bar lives below
    it, not inside it.

## Complexity

`small`-`medium` — 1-2 AOM sessions. Largest risk: runtime
detection without spawning subprocesses for every cwd change.

## Open questions

- **Runtime detection strategy**: file-based heuristics
  (`package.json` → node, `Cargo.toml` → rust, `pyproject.toml`
  → python, `go.mod` → go, `Gemfile` → ruby) are fast and
  subprocess-free. Should we *also* run the actual binary to get
  the exact installed version, or just show the file-declared
  version? File-based is simpler and almost always correct;
  prefer that. ESCALATE if user wants binary probing.
- **Cache invalidation**: cwd-keyed cache works for git/branch
  but a branch can change *while in the same cwd*. Refresh on
  every `CwdChanged`; do NOT add a filesystem watcher.
- **Multiple runtimes detected**: a polyglot repo
  (`package.json` + `Cargo.toml`) — show both? show first? show
  the one matching last command run? v1 = show first detected,
  ordered by language popularity in the repo top-level.
- **Visibility in AFK**: confirmed out of scope above; ESCALATE
  if the user later wants it.
