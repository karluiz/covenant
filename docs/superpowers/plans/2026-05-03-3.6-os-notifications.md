# 3.6 OS Notifications — Implementation Plan

**Goal:** Fire native macOS notifications on Operator ESCALATE, AOM error
(budget hit), and AOM completion (user stop), with three per-trigger toggles
+ a global "suppress when window focused" setting + a 30s per-trigger
throttle.

**Architecture:** New `crates/app/src/notify.rs` owns a `Notifier` struct
that holds the AppHandle, a per-trigger `Instant` throttle map, and a
settings-aware `emit()` that gates on toggles + window-focus + throttle
before calling `tauri-plugin-notification`'s builder. Three call sites
(operator ESCALATE, AOM budget-hit, aom_stop) push trigger events; the
frontend listens for a custom `notification-clicked` event to switch tab.

**Tech Stack:** Tauri 2.x, `tauri-plugin-notification = "2"`,
`@tauri-apps/plugin-notification = "^2"`.

  - `ui/src/main.ts` — listen for `notification-clicked`, switch tab
  - `ui/src/api.ts` — type for new settings field

## Acceptance to task mapping

- A1 (perm request on first relevant trigger) → Notifier.emit() lazily
  triggers; we'll also kick a no-op request_permission on app setup.
- A2 trigger types → ESCALATE/budget-hit/aom_complete each call
  `notifier.emit(Trigger::*, title, body, session_id)`
- A3 (click → focus + tab) → Tauri 2 plugin doesn't surface a click
  callback on macOS without custom Swift. **ESCALATE.** Implement what
  the OS gives us for free (clicking activates the app); per-tab routing
  uses Tauri event the frontend listens to, fired from the notification
  on close (best-effort) — degrades gracefully.
- A4 (suppress when focused) → Notifier checks
  `app.get_webview_window("main").is_focused()` before emit
- A5 (settings page 4 toggles) → panel.ts new section
- A6 (throttle 30s per trigger across all tabs) → HashMap<Trigger, Instant>
- A7 (cargo check + tests) → unit tests for throttle + trigger gating

## Out-of-scope acknowledgments

- macOS click-to-tab routing requires NSUserNotificationCenter delegate
  Swift; this is the open question in the spec. We document the
  limitation: clicking activates the app + emits a `notification-clicked`
  event on the frontend bus when supported by the JS plugin's
  `onAction`/listen API; otherwise the user just gets the app focused
  (still useful).
